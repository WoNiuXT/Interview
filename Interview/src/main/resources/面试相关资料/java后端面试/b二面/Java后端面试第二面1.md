1\. 问题：你了解 G1 垃圾收集器吗？相比 CMS，它有哪些进步？



出题意图：考察对现代垃圾收集器（特别是 G1）的深入理解，以及对 GC 技术演进方向的认知。



参考答案 (源自《Java 虚拟机面试题速记通关版》)：



G1 (Garbage-First) 收集器：是 JDK 9 以后的默认垃圾收集器，专为多核处理器和大内存机器设计。它将整个堆划分为多个大小相等的 Region，但逻辑上仍分代。



相比 CMS 的进步：



可预测的停顿时间：G1 可以设定一个预期的停顿时间目标 (-XX:MaxGCPauseMillis)。它会根据这个目标，优先回收垃圾最多的 Region（Garbage-First 名字的由来），从而在有限时间内回收尽可能多的空间。

避免内存碎片：G1 整体上基于 标记-整理 (Mark-Compact) 算法，在回收过程中会进行对象复制和整理，解决了 CMS 使用 标记-清除 (Mark-Sweep) 算法产生的内存碎片问题，减少了 Full GC 的风险。

并发更高效：G1 使用 Remembered Set (RSet) 来记录跨 Region 的引用，避免了像 CMS 那样在并发标记阶段扫描整个老年代。回收时，只需关注 RSet 中记录的 Region。

处理大对象：G1 有专门的 Humongous Region 来存储超过 Region 大小一半的大对象，避免了大对象频繁在老年代分配和移动。

评分要点：



5分：能清晰说出 G1 的 Region 划分和逻辑分代概念，并能从停顿可预测、内存整理、RSet 效率、大对象处理四个方面详细对比 CMS 的优势。

3\. 问题：你了解 AQS（AbstractQueuedSynchronizer）吗？它是如何实现线程同步的？



出题意图：考察对 Java 并发包（JUC）基石的理解深度。



参考答案 (源自《Java 并发面试题速记通关版》)：



AQS 是什么：AQS 是 java.util.concurrent 包中用于构建锁和同步器（如 ReentrantLock、Semaphore、CountDownLatch）的基础框架。



核心原理：



状态管理：内部有一个 volatile int state 变量，用于表示同步状态。例如在 ReentrantLock 中，state 表示锁的重入次数。

CLH 等待队列：一个 FIFO 的双向链表队列。当线程获取同步状态失败时，AQS 会将当前线程和等待状态等信息构造成一个 Node 节点，并加入到队列尾部，同时阻塞该线程。

CAS 操作：对 state 的修改和对队列的操作大量依赖 CAS（如 compareAndSetState），保证了原子性和无锁并发安全性。

模板方法模式：AQS 定义了获取和释放同步状态的模板方法（如 acquire()、release()），而将具体的获取/释放逻辑（如 tryAcquire()、tryRelease()）留给子类实现，让子类可以灵活定义同步规则。

7\. 问题：请简述 RocketMQ 事务消息的实现原理。



出题意图：考察对分布式事务中“最终一致性”方案的深入理解。



参考答案 (源自《消息队列面试题速记通关版》)：



核心思想：半消息 + 事务回查。目的是解决本地数据库操作和发送消息这两件事的原子性问题。



流程：



发送半消息：生产者向 Broker 发送一个“半消息”，该消息对消费者不可见，Broker 暂存。

执行本地事务：生产者成功发送半消息后，执行本地数据库事务（如扣减库存）。

提交/回滚消息：根据本地事务的执行结果，生产者向 Broker 提交 Commit 或 Rollback 指令。如果收到 Commit，Broker 使消息对消费者可见；如果收到 Rollback，Broker 删除半消息。

事务回查：如果生产者发送完半消息后，因为宕机等原因，没有向 Broker 发送最终指令，Broker 会定时向生产者发起事务状态回查。生产者需要实现一个检查接口，根据本地事务的执行状态（是否提交）告诉 Broker 是提交还是回滚该消息。

评分要点：



5分：能清晰、完整地阐述事务消息的“半消息”和“事务回查”两个核心机制，并能画出流程图。

9\. 问题：请介绍一下 @Transactional 注解的失效场景。



出题意图：考察对 Spring 事务管理原理（AOP 代理）的深入理解。



参考答案 (源自《SpringBoot 面试题速记通关版》)：



核心原因：Spring 的事务管理基于 AOP 代理实现，失效的根本原因是没有通过代理对象调用被 @Transactional 注解的方法。



常见失效场景：



同一个类中方法调用：最经典的失效场景。如 ServiceA 的方法 a() 没有加事务，内部调用加了 @Transactional 的方法 b()。这相当于 this.b()，调用的是目标对象本身的方法，而不是 Spring 生成的代理对象，因此事务不起作用。

方法访问权限问题：@Transactional 注解的方法不能是 private 的，必须是 public 或 protected（如果使用代理类的方式）。因为代理类无法覆盖 private 方法。

异常类型问题：默认情况下，Spring 事务只在遇到运行时异常 (RuntimeException) 时回滚。如果方法抛出的是受检异常 (Checked Exception)，如 SQLException，事务不会回滚（除非通过 rollbackFor 属性指定）。

方法被 final 修饰：如果方法被 final 修饰，子类（代理类）无法重写该方法，导致 AOP 增强失效。

传播属性配置不当：例如设置为 Propagation.NOT\_SUPPORTED，表示以非事务方式运行，自然不会生效。

评分要点：



5分：能明确指出失效的根本原因是“没有通过代理对象调用”，并能系统性地列举出 4-5 种失效场景，对每种场景都能解释清楚原因。

10\. 问题：你如何理解分布式锁？基于 Redis 的实现需要注意哪些问题？



出题意图：考察对分布式协调的基础概念和常见实现方案的深入理解，特别是其局限性。



参考答案 (源自《Redis 面试题速记通关版》)：



分布式锁定义：在分布式系统中，多个节点对共享资源进行互斥访问的一种协调机制。



Redis 实现的核心命令：SET key value NX PX milliseconds。



NX：只在键不存在时设置，保证只有一个客户端能成功。



PX：设置锁的过期时间，防止客户端宕机导致死锁。



value：通常使用 UUID 等唯一标识，确保只有锁的持有者才能释放锁。



需要注意的问题：



锁过期时间设置：业务执行时间不确定，锁过期时间短了会导致提前释放，其他线程可进入；长了又会影响响应。需要结合看门狗 (Watchdog) 机制（如 Redisson 提供）来自动续期。

释放锁的原子性：不能简单地先 get 判断是否是自己加的锁，再 del。因为这两个操作非原子，可能导致误删别人刚获得的锁。必须使用 Lua 脚本 将“判断+删除”操作原子化。

lua

if redis.call("get", KEYS\[1]) == ARGV\[1] then

&nbsp;   return redis.call("del", KEYS\[1])

else

&nbsp;   return 0

end

主从切换导致锁失效：这是 Redis 分布式锁的一个重大隐患。当主节点写入锁后，还没来得及同步到从节点就宕机了，从节点被提升为主节点，此时新的客户端就可以再次获取同一把锁，导致锁失效。Redlock 算法试图解决此问题，但其本身也复杂且存在争议。

评分要点：



5分：能准确描述分布式锁的概念，能详细说明 Redis 实现的核心命令和释放锁的 Lua 脚本。能深入、系统地指出三大潜在问题（过期时间、原子性、主从失效），并能提出相应的解决思路（看门狗、Lua、Redlock 的权衡）。

11\. 问题：MySQL 的 MVCC（多版本并发控制）是如何实现的？它解决了什么问题？



出题意图：考察对 InnoDB 核心并发控制机制的深入理解。



参考答案 (源自《MySQL 面试题速记通关版》)：



解决的问题：MVCC 主要解决了读写之间的阻塞问题，在 READ COMMITTED 和 REPEATABLE READ 隔离级别下，让读操作不加锁也能读到一致的数据快照，极大提升了并发读性能。



实现原理：依赖于三个隐藏字段和 undo log。



隐藏字段：每行记录都有两个重要隐藏字段：

DB\_TRX\_ID：创建或最后一次修改该行记录的事务 ID。

DB\_ROLL\_PTR：回滚指针，指向该行记录的上一个版本（存储在 undo log 中）。多个版本通过指针形成一个版本链。

undo log：当数据被修改时，旧版本的数据会被记录到 undo log 中，并通过 DB\_ROLL\_PTR 串联起来。

Read View (读视图)：当一个事务执行快照读（普通的 SELECT）时，会生成一个 Read View。它记录了当前活跃的事务 ID 列表。

可见性判断：根据 Read View 和行记录中的 DB\_TRX\_ID，按照特定规则（例如：只能看到比自己早的事务 ID 的修改，看不到自己之后的事务和未提交的事务的修改）判断该版本的可见性。如果当前版本不可见，就通过 DB\_ROLL\_PTR 找到上一个版本，直到找到可见的版本。

评分要点：



5分：能清晰说出 MVCC 解决的核心问题。能完整、准确地描述其实现依赖的三个要素（隐藏字段、undo log、Read View），并能简要说明可见性判断规则。



4分：能说出 MVCC 的作用，能讲出隐藏字段和 undo log 的作用，但对 Read View 和判断规则理解不清。



3分：知道 MVCC 用于提高并发，但对其实现原理模糊。



2分：完全不了解 MVCC。



12\. 问题：你遇到过 MySQL 死锁吗？你是如何排查和解决的？



出题意图：考察对数据库锁的实战经验和问题排查能力。



参考答案 (源自《MySQL 面试题速记通关版》)：



排查方法：



查看死锁日志：执行 SHOW ENGINE INNODB STATUS\\G，查看 LATEST DETECTED DEADLOCK 部分。日志会清晰地指出哪些事务（TRANSACTION）持有了什么锁（lock struct），正在等待什么锁，以及导致死锁的 SQL 语句。

分析事务和锁表：可以查询 information\_schema.INNODB\_TRX 表查看当前正在执行的事务，INNODB\_LOCKS 和 INNODB\_LOCK\_WAITS 表查看锁和等待关系。

解决方案：



调整代码逻辑，保持一致的加锁顺序：这是最根本的解决方案。例如，两个事务都先更新表 A，再更新表 B，而不是一个先 A 后 B，另一个先 B 后 A。

缩短事务长度：尽量减少事务中不必要的操作，特别是将外部调用、文件读写等慢操作移出事务。

选择合适的隔离级别：评估业务是否能接受较低的隔离级别，如 READ COMMITTED，它可能减少锁冲突。

优化索引：确保 UPDATE 和 DELETE 语句能高效地使用索引，避免扫描大量行导致锁范围过大（从行锁升级为表锁或间隙锁）。

评分要点：



5分：能准确说出通过 SHOW ENGINE INNODB STATUS 查看死锁日志，并能读懂日志内容。能清晰、逻辑地分析死锁原因，并提出多种解决方案（如统一加锁顺序、缩短事务等），并能举例说明。



4分：能说出用 SHOW ENGINE INNODB STATUS 查看，能分析简单死锁原因，但解决方案不够全面。



3分：知道有死锁这回事，但不知道如何排查，或解决方案停留在概念层面。



2分：完全没有排查和处理死锁的经验。



13\. 问题：假设一张表有 5000 万数据，你是如何进行数据归档或删除的？

核心原则：避免一次性操作大量数据，防止产生大事务、长锁和主从延迟。



方案（分批删除）：



选择删除条件：通常根据主键或时间戳范围进行分批。例如，按 id 范围删除。

编写分批删除脚本：

sql

\# 每次删除 1000 条记录

DELETE FROM large\_table

WHERE id BETWEEN 1 AND 1000;

\# 然后根据主键顺序，继续删除下一批

添加休眠：在每批删除之间，让程序 sleep 一小段时间（如 1 秒），减轻对数据库的冲击。

使用工具：可以使用 pt-archiver 工具，它是专门为此类任务设计的，可以以可控的速度、低负载地将数据迁移到另一个表或直接删除。

数据归档：通常是 INSERT INTO archive\_table SELECT ... FROM large\_table WHERE ...，然后再删除原表数据。也可以使用在线 DDL 工具或 pt-archiver 来完成。



评分要点：



5分：能深刻意识到大事务的风险，并清晰阐述分批处理的原则。能提供具体的分批策略、代码示例和工具推荐，并能解释为什么需要这样做。



4分：能说出需要分批处理，能提供基本的思路，但对细节和工具使用不够熟悉。



3分：知道不能直接 DELETE，但说不出具体可行的分批方案。

15\. 问题：设计一个短链系统，你会如何设计？



出题意图：考察候选人的抽象能力、数据存储选型和性能优化思维。



参考答案 (源自《后端系统设计面试题速记通关版》)：



核心功能：将长 URL 转换为一个唯一的短字符串，并通过短字符串访问时能 302 重定向到长 URL。



核心难点：短码的生成和存储。



设计方案：



短码生成：使用发号器生成一个自增的唯一 ID（如通过数据库号段模式或 Snowflake 算法）。然后将该 ID 转换为 62 进制（a-zA-Z0-9），得到一个 6-8 位的短码。这种方式生成的短码趋势递增，且不重复。

存储设计：

读多写少：访问一个短链的请求量远大于生成量。因此必须引入缓存。

存储选型：数据库选用 MySQL，表结构为 (id, short\_code, original\_url, expire\_time)。在 short\_code 上建立唯一索引。

缓存层：使用 Redis。当用户访问短链时，先查 Redis。如果命中，直接返回原 URL；如果未命中，则查数据库，查到后回写 Redis 并设置过期时间。这样可以扛住巨大的读流量。

重定向：使用 302 临时重定向，便于统计点击量，而不是 301 永久重定向。

扩展功能：统计点击量。可以通过在 Redis 中维护计数，然后异步（如定时任务或 Kafka）将数据批量刷入数据库或分析系统。

评分要点：



5分：能清晰地阐述短链系统的核心流程（生成、存储、访问）。能深入讨论短码生成的“发号器+62进制”方案，并指出其优势。能结合“读多写少”的特性，合理设计“数据库+Redis缓存”的存储架构。



4分：能说出短码生成和存储的基本思路，但对发号器细节和缓存策略考虑不够周全。



3分：知道要生成短码，但方案不明确；知道要存储，但对读写特性分析不足。



2分：设计思路混乱，无法形成有效方案。
