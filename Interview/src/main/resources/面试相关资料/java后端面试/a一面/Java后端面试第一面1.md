1\. 问题：请解释一下 ArrayList 和 LinkedList 的区别，它们分别适用于什么场景？



出题意图：考察对常用集合类底层数据结构的理解，以及根据场景做技术选型的能力。



参考答案 (源自《Java 面试题速记通关版》)：



底层结构：ArrayList 底层是基于 动态数组 实现的；LinkedList 底层是基于 双向链表 实现的。



查询效率：ArrayList 支持随机访问，get(int index) 时间复杂度为 O(1)。LinkedList 需要从链表头/尾遍历，时间复杂度为 O(n)。



增删效率：ArrayList 在中间插入或删除元素时，需要移动后续所有元素（System.arraycopy），平均时间复杂度 O(n)。LinkedList 只需要修改前后节点的指针，时间复杂度为 O(1)，但前提是已经定位到该位置（定位本身需要 O(n)）。



适用场景：



ArrayList 适用于读多写少、频繁按索引访问的场景，如大多数查询和遍历操作。



LinkedList 适用于频繁在头部或尾部插入/删除，且不经常按索引访问的场景，如实现队列（Queue）或栈（Stack）。

2\. 问题：HashMap 的底层实现原理是什么？JDK 1.8 中引入了红黑树，为什么？



出题意图：考察对最核心集合的源码级理解，以及为解决特定问题（性能退化）而进行的优化。



参考答案 (源自《Java 集合面试题速记通关版》)：



底层结构：HashMap 在 JDK 1.8 中的底层是 数组 + 链表/红黑树。



计算索引：先计算 key 的 hashCode()，经过扰动函数（高16位与低16位异或）使散列更均匀，再通过 (n - 1) \& hash 计算出元素在数组（称为桶，bucket）中的位置。



处理冲突：当多个元素定位到同一个桶时，以链表形式存储。



引入红黑树的原因：为了解决哈希冲突严重时，链表过长导致查询效率退化的问题。



触发条件：当链表长度大于阈值 8，并且数组长度大于 64 时，链表会转换为红黑树。红黑树的查询时间复杂度为 O(log n)，远优于链表的 O(n)，保证了在最坏情况下的性能。



退化条件：当红黑树节点数量减少到 6 时，又会转换回链表，以节省空间和应对少量节点的场景。

5分：能详细说明数组+链表/红黑树的结构，讲清楚索引计算过程，并准确说出树化阈值（8）和退化阈值（6）及数组长度限制（64），解释其目的是为了防止性能退化。



4分：能说出数组+链表/红黑树，知道链表过长会转红黑树，但对具体阈值和条件记忆不清。



3分：能说出 HashMap 的基本存储方式（键值对，数组+链表），但不知道红黑树。



2分：对 HashMap 结构模糊，或将其与 Hashtable 混淆。



3\. 问题：ConcurrentHashMap 是如何实现线程安全的？JDK 1.7 和 1.8 的实现有什么区别？



出题意图：考察对并发容器的深入理解，以及对技术演进背后原因的认识。



参考答案 (源自《Java 集合面试题速记通关版》)：



JDK 1.7 (分段锁)：



底层采用 Segment 数组 + HashEntry 数组。



Segment 继承自 ReentrantLock，每个 Segment 守护着其下的多个桶。理论上最大并发度等于 Segment 的数量（默认 16）。



这是一种锁分段技术，降低了锁的粒度。



JDK 1.8 (CAS + synchronized)：



摒弃了 Segment，改用 Node 数组 + 链表/红黑树。



锁粒度进一步降低，只对数组的每个桶（即链表的头节点或红黑树的根节点）加锁。



在写入时，如果对应桶为空，则通过 CAS 原子操作进行插入，避免加锁；如果桶不为空，则对桶头节点使用 synchronized 锁定。



读操作 get() 无锁，通过 volatile 保证可见性。



支持多线程协同扩容，效率更高。



评分要点：



5分：能清晰对比 1.7 的分段锁（Segment）和 1.8 的 CAS + synchronized 头节点锁，能解释锁粒度、并发度、以及读操作无锁和多线程扩容等优化点。



4分：能说出 1.8 主要用 CAS + synchronized 锁头节点，但对 1.7 的分段锁细节和两种版本的性能优劣对比不够深入。



3分：能说出它是线程安全的，并发度比 Hashtable 高，但无法解释具体实现原理。



2分：将 ConcurrentHashMap 的同步方式与 Hashtable 的 synchronized 方法混淆，或完全不知道区别。



4\. 问题：Java 中 == 和 equals() 方法有什么区别？重写 equals() 时为什么必须重写 hashCode()？



出题意图：考察对对象相等性判断和哈希表原理的理解。



参考答案 (源自《Java 面试题速记通关版》)：



== vs equals()：



== 是运算符。对于基本类型，比较的是值；对于引用类型，比较的是对象在内存中的引用地址。



equals() 是 Object 类的方法，默认实现也是通过 == 比较引用地址。但它的设计意图是让子类重写，以实现逻辑相等的比较。



重写 equals() 必须重写 hashCode()：



这是为了遵守 Object 类的约定：如果两个对象根据 equals() 方法是相等的，那么它们的 hashCode() 必须相等。



以 HashMap 为例，它先通过 hashCode() 定位到桶，再用 equals() 比较桶内的元素。如果只重写 equals() 而不重写 hashCode()，两个逻辑相等的对象（例如两个 id 相同的 User 对象）可能计算出不同的哈希码，导致它们被放入不同的桶中，那么当我们用其中一个作为 key 去 get() 另一个时，就永远找不到了。



评分要点：



5分：能清晰解释 == 在基本类型和引用类型的区别，能指出 equals() 的默认行为。能准确说明 hashCode 和 equals 的约定，并能结合 HashMap 的工作原理（先定位桶，再比较元素）解释为何必须同时重写。



4分：能说清 == 和 equals() 的区别，也能说出重写 equals 必须重写 hashCode，但无法结合集合类原理深入解释原因。



3分：能说出 == 比较地址，equals() 比较内容，但不知道为什么必须重写 hashCode。



2分：概念不清，混淆两者。

设计理念：



抽象类：强调 “是什么” (is-a) 的关系，为一系列具有共同属性和行为的类提供代码复用和模板。



接口：强调 “能做什么” (has-a / like-a) 的能力，定义了一组契约或规范，用于实现类之间的解耦和多态。

构造器：抽象类有构造器，用于初始化；接口没有构造器。

7\. 问题：Java 中如何判断一个对象可以回收？有哪些垃圾回收算法？



出题意图：考察 GC 的核心原理和不同回收算法的优缺点。



参考答案 (源自《Java 面试题速记通关版》 \& 《Java 虚拟机面试题速记通关版》)：



判断方法：主流 JVM (HotSpot) 采用 可达性分析算法。从 GC Roots（如虚拟机栈引用的对象、静态属性引用的对象、常量引用的对象等）出发，通过引用链向下搜索。如果一个对象到 GC Roots 没有任何引用链相连，则该对象被判定为可回收。



垃圾回收算法：



标记-清除 (Mark-Sweep)：先标记所有存活对象，再统一回收未被标记的对象。缺点是会产生内存碎片。



复制 (Copying)：将内存分为两块，只使用一块。当这块用完，将存活对象复制到另一块，再一次性清理整块。优点是实现简单、无碎片，缺点是内存利用率低。常用于新生代（Eden 区与 Survivor 区）。



标记-整理 (Mark-Compact)：标记过程与“标记-清除”一样，但后续不是直接清理，而是将所有存活对象向一端移动，然后清理掉边界以外的内存。无碎片，但整理有开销。常用于老年代。



评分要点：



5分：能准确说出可达性分析算法和 GC Roots 的概念，并能清晰对比三种回收算法的原理、优缺点（碎片、效率、空间利用率）。



4分：能说出可达性分析，能说出两种算法（如复制和标记-清除），但对第三种和对比分析不够清晰。



3分：能说出引用计数或可达性分析，但对回收算法理解模糊。



2分：概念错误，如认为通过计数器判断，或算法混淆。



8\. 问题：Java 中的强引用、软引用、弱引用和虚引用有什么区别？



出题意图：考察对引用类型及其在 GC 中不同行为的理解，这是实现缓存等高级功能的基础。



参考答案 (源自《Java 虚拟机面试题速记通关版》)：



强引用：最常见的引用，如 Object obj = new Object()。只要强引用还存在，垃圾收集器永远不会回收被引用的对象。



软引用 (SoftReference)：描述一些还有用但非必需的对象。在系统将要发生内存溢出异常之前，会把这些对象列入回收范围进行第二次回收。适合做缓存。



弱引用 (WeakReference)：描述非必需对象。被弱引用关联的对象只能生存到下一次垃圾收集发生为止。典型应用是 ThreadLocal 的 Entry，其 key 就是弱引用，防止 ThreadLocal 对象无法被回收导致内存泄漏。



虚引用 (PhantomReference)：最弱的引用关系。为一个对象设置虚引用关联的唯一目的是能在这个对象被回收时收到一个系统通知，通常用于管理堆外内存。



评分要点：



5分：能清晰区分四种引用，准确描述其回收时机，并能结合实际应用举例（如软引用做缓存，弱引用与 ThreadLocal）。



4分：能说出四种引用和它们的回收强度，但对应用场景描述不够准确。



3分：能说出强、软、弱三种引用，但对虚引用和它们的核心区别（回收时机）模糊。



2分：概念混淆，说不清区别。

9\. 问题：synchronized 和 ReentrantLock 有什么区别？



出题意图：考察对两种常用锁的理解，包括它们的实现层级、特性和适用场景。



参考答案 (源自《Java 并发面试题速记通关版》)：



实现层级：synchronized 是 JVM 层面 的内置锁，通过 monitorenter / monitorexit 指令实现；ReentrantLock 是 JDK 层面 (API) 的锁。



锁释放：synchronized 在退出同步块或方法时自动释放；ReentrantLock 必须手动释放，通常需要在 finally 块中调用 unlock()。



功能特性：



等待可中断：ReentrantLock 可以通过 lockInterruptibly() 让等待锁的线程响应中断；synchronized 不行。



公平锁：ReentrantLock 可以通过构造参数 true 指定为公平锁（按照线程等待时间获取锁）；synchronized 只能是非公平锁。



绑定多个条件：ReentrantLock 可以创建多个 Condition 对象，实现更精细的线程等待/唤醒；synchronized 只能配合 wait()/notify()。



性能：早期 synchronized 性能较差，但经过 JDK 1.6 的锁升级（偏向锁 -> 轻量级锁 -> 重量级锁）优化后，与 ReentrantLock 差距不大。



评分要点：



5分：能清晰对比两者的实现层级，并完整列出 ReentrantLock 的三大高级特性（中断、公平锁、Condition），并能提及性能优化。



4分：能说出主要区别，如手动释放、支持公平锁，但可能遗漏 Condition 或对锁升级不了解。



3分：能说出 synchronized 是自动的，ReentrantLock 是手动的，但对高级特性了解不多。



2分：概念混淆，不知道 ReentrantLock 的存在，或认为 synchronized 可以公平锁。



10\. 问题：volatile 关键字的作用是什么？它为什么不能保证原子性？



出题意图：考察对 volatile 核心概念（可见性、有序性）的精确理解。



参考答案 (源自《Java 并发面试题速记通关版》)：



作用：



保证可见性：一个线程对 volatile 变量的修改，会立刻被强制写回主内存；其他线程读取该变量时，会强制从主内存中读取，从而保证了修改对其他线程立即可见。



禁止指令重排序：通过内存屏障，保证 volatile 变量前后的指令不会被编译器或处理器重排序，从而保证了程序执行的顺序性。



为什么不能保证原子性：



原子性是指一个操作要么全部执行成功，要么全部不执行，中途不能被中断。例如 count++ 这个操作，在字节码层面是分为“读-改-写”三步的，不是一个原子操作。



volatile 只保证了每次读取时都能读到最新值，但无法保证“读-改-写”这三步的不可分割性。当多个线程同时执行 count++ 时，可能发生数据覆盖，导致最终结果小于期望值。



评分要点：



5分：能准确说出 volatile 的可见性和禁止重排序两大作用。能深刻理解并举例说明原子性的概念，并用“读-改-写”三步清晰解释为何 volatile 不能保证原子性。



4分：能说出可见性和禁止重排序，能说出 i++ 不是原子的，但解释不够深入。



3分：能说出可见性，但无法解释指令重排，对原子性问题的解释模糊。



2分：将 volatile 等同于锁，认为它能保证原子性，概念错误。



11\. 问题：Java 线程池 ThreadPoolExecutor 的核心参数有哪些？请简述其工作流程。



出题意图：考察对线程池核心机制的理解，这是高并发编程的基础。



参考答案 (源自《Java 并发面试题速记通关版》)：



核心参数：



corePoolSize：核心线程数，即使空闲也会保留。

maximumPoolSize：线程池允许的最大线程数。

keepAliveTime：非核心线程空闲存活时间。

unit：存活时间的时间单位。

workQueue：阻塞队列，用于存放等待执行的任务。

threadFactory：线程工厂，用于创建新线程。

handler：拒绝策略，当任务太多无法处理时的处理方式。

工作流程：



提交任务后，如果当前线程数 < corePoolSize，则直接创建核心线程执行任务。

如果线程数 >= corePoolSize，则将任务放入 workQueue 中等待。

如果队列已满，且当前线程数 < maximumPoolSize，则创建非核心线程执行任务。

如果队列已满，且线程数已达到 maximumPoolSize，则触发拒绝策略。

评分要点：



5分：能准确说出全部 7 个参数，并能清晰、有条理地描述线程池处理任务的全流程。



4分：能说出核心参数，能大致描述流程，但可能遗漏 threadFactory 或对流程步骤顺序记忆不清。



3分：能说出 3-4 个参数，对流程理解模糊。



2分：基本不知道线程池参数，或流程完全错误。



12\. 问题：CountDownLatch 和 CyclicBarrier 有什么异同？



出题意图：考察对并发工具类使用场景和原理的辨析。



参考答案 (源自《Java 并发面试题速记通关版》)：



相同点：都能实现线程间的等待。



不同点：



计数器用法：



CountDownLatch 是一个线程（或多个）等待其他多个线程完成某件事。计数器只能使用一次，是“一等多”。



CyclicBarrier 是多个线程互相等待，直到所有线程都到达某个同步点，再一起继续执行。计数器可以被重置重用，是“多等多”。



核心方法：



CountDownLatch 使用 countDown() 减计数，await() 阻塞等待计数为 0。



CyclicBarrier 使用 await() 让线程等待，当最后一个线程调用 await() 时，所有等待线程被唤醒。



作用场景：CountDownLatch 用于启动服务前等待多个依赖服务加载完成；CyclicBarrier 用于并行计算中，等待所有子任务计算完成后合并结果。



评分要点：



5分：能清晰对比两者的核心区别（计数器能否重用，是一等多还是多等多），并能举例说明各自的应用场景。



4分：能说出主要区别，但可能混淆两个概念，或场景举例不恰当。



3分：能知道它们是用来做线程同步的，但无法清晰区分。



2分：完全混淆，说不出异同。

13\. 问题：MySQL 的 InnoDB 引擎中，聚簇索引和非聚簇索引有什么区别？什么是回表查询？



出题意图：考察对 InnoDB 索引结构的核心理解，这是 SQL 优化的基础。



参考答案 (源自《MySQL 面试题速记通关版》)：



聚簇索引：叶子节点直接存储完整的数据行。InnoDB 表中，主键索引就是一个聚簇索引。每张表只能有一个聚簇索引，因为数据行只能按一种物理顺序存放。



非聚簇索引（二级索引）：叶子节点存储的是索引列的值 + 主键值。



回表查询：当使用二级索引进行查询时，会先在二级索引的 B+ 树上找到对应的主键值，然后再拿着这个主键值到聚簇索引的 B+ 树上查找完整的行记录。这个过程称为“回表”。



评分要点：



5分：能清晰对比两者叶子节点存储内容的不同，并能准确描述“回表”的完整过程。



4分：能说出聚簇索引存数据，二级索引存主键，能说出“回表”概念，但过程描述不清晰。



3分：能说出有这两种索引，但对存储内容和查询过程理解模糊。



2分：概念不清，将两者混淆。

15\. 问题：事务的四大特性（ACID）分别是什么？MySQL 的 InnoDB 是如何实现它们的？



出题意图：考察对事务本质的理解以及存储引擎层面的实现机制。



参考答案 (源自《MySQL 面试题速记通关版》)：



原子性 (Atomicity)：事务是一个不可分割的原子单元。通过 undo log 实现。如果事务失败，可以通过 undo log 回滚到事务开始前的状态。



一致性 (Consistency)：事务执行前后，数据库的完整性约束没有被破坏。由其他三个特性共同保证，同时数据库自身（如约束、触发器）也参与保证。



隔离性 (Isolation)：多个事务并发执行时，一个事务的执行不应被其他事务干扰。通过 锁机制 和 MVCC (多版本并发控制) 实现。



持久性 (Durability)：事务一旦提交，其结果就是永久性的。通过 redo log 实现。事务提交时，会先将修改记录到 redo log 并刷盘，保证即使宕机，重启后也能通过 redo log 恢复数据。



评分要点：



5分：能完整说出 ACID 四个特性，并能准确指出 undo log 用于原子性，redo log 用于持久性，锁和 MVCC 用于隔离性。



4分：能说出 ACID 和对应的实现技术，但可能会把 undo log 和 redo log 的作用说反。



3分：能说出 ACID 的定义，但对 InnoDB 的实现方式不了解。



2分：概念不清，不知道 ACID 是什么。

16\. 问题：请简述 TCP 三次握手的过程？为什么需要三次握手而不是两次？



出题意图：考察对可靠连接建立过程的理解。



参考答案 (源自《计算机网络面试题速记通关版》)：



过程：



客户端 -> 服务端：发送 SYN 报文，并进入 SYN\_SENT 状态。

服务端 -> 客户端：收到 SYN，回复 SYN-ACK 报文，并进入 SYN\_RCVD 状态。

客户端 -> 服务端：收到 SYN-ACK，回复 ACK 报文，双方进入 ESTABLISHED 状态。

需要三次的原因：



确认双方收发能力：第一次握手，服务端确认客户端能发；第二次握手，客户端确认服务端能收能发；第三次握手，服务端确认客户端能收。只有三次才能确保双方都具备收发能力。

防止失效的连接请求：如果只用两次，当客户端发送的第一个 SYN 在网络中滞留，客户端超时重发 SYN 并成功建立连接、传输数据、释放连接后，滞后的那个 SYN 才到达服务端，服务端会认为是一个新连接请求，并发送 SYN-ACK，等待客户端数据，造成资源浪费。

评分要点：



5分：能准确描述三次握手的过程和状态变化，并能从“确认双方收发能力”和“防止失效请求”两个角度深刻解释为什么需要三次。



4分：能说清过程，能解释“确认能力”，但对“防止失效请求”的解释不够清晰。



3分：能大致说出过程，但无法解释原因。



2分：过程描述错误。



17\. 问题：TIME\_WAIT 状态出现在 TCP 的哪个阶段？它的作用是什么？



出题意图：考察对 TCP 连接关闭细节的理解，以及其对高并发服务的影响。



参考答案 (源自《计算机网络面试题速记通关版》)：



出现阶段：出现在主动关闭连接的一方，在四次挥手的最后一个 ACK 报文发出后，会进入 TIME\_WAIT 状态，持续时间为 2MSL（最大报文段生存时间）。



作用：

让旧连接的报文在网络中消失：防止当前连接关闭后，网络中延迟的旧连接数据包被后续使用相同四元组（源IP、源端口、目标IP、目标端口）的新连接误认为是有效数据。

评分要点：



5分：能准确说出 TIME\_WAIT 出现在主动关闭方，并能清晰阐述两大作用（保活最后一个ACK和让旧报文消失）。

19\. 问题：什么是缓存穿透、缓存击穿和缓存雪崩？如何解决？



出题意图：考察对缓存常见问题的认知和应对策略，这是保证缓存系统稳定性的关键。



参考答案 (源自《Redis 面试题速记通关版》)：



缓存穿透：查询一个不存在的数据，导致每次请求都直接打到数据库。



解决：布隆过滤器，将所有可能存在的数据哈希到一个足够大的 bitmap 中，一个不存在的请求会被这个 bitmap 拦截。或者缓存空对象（即使是 null 也缓存，并设置短过期时间）。



缓存击穿：一个热点 key 在失效的瞬间，大量请求同时打到数据库。



解决：互斥锁 (Mutex)。只让一个线程去数据库加载数据并重建缓存，其他线程等待。或设置热点数据永不过期，由后台线程异步更新。



缓存雪崩：大量 key 同时失效，或者缓存节点故障，导致大量请求涌入数据库。



解决：给 key 的过期时间添加一个随机值，避免集体失效。采用高可用缓存架构（如 Redis 哨兵或集群）。本地缓存 + Redis 二级缓存作为兜底。



评分要点：



5分：能清晰区分三种现象的本质，并针对性地给出多种解决方案，并能解释原理（如布隆过滤器、互斥锁）。



4分：能说出三种现象和对应的主要解决方案，但对细节和原理解释不够深入。



3分：能大致区分现象，但解决方案零散或不完整。



2分：概念混淆，将三者混为一谈。



20\. 问题：如何保证缓存与数据库的数据一致性？请介绍一种常用的方案。



出题意图：考察对缓存和数据库双写一致性问题复杂性的认知，以及解决思路。



参考答案 (源自《Redis 面试题速记通关版》)：



推荐方案：先更新数据库，再删除缓存 (Cache Aside Pattern)。



为什么推荐：这是实践中最常用且相对安全的方案。在高并发下，虽然存在一个极短的时间窗口（在更新数据库后、删除缓存前），读请求可能会读到旧数据（因为缓存未删除），但下次读请求就能读到新数据。这个不一致窗口很小，通常可以接受。



操作流程：



写请求：先更新数据库。

写请求：成功后删除缓存。

读请求：先读缓存，如果没有，则从数据库读取，并写入缓存。

为什么要删除而不是更新缓存：避免复杂的并发更新逻辑。删除操作是幂等的，下次读请求自然会拉取最新数据。



评分要点：



5分：能清晰阐述“先更新DB，再删缓存”的方案，能解释为什么这是常用方案（不一致窗口小），并能说明为什么删缓存比更新缓存好。



4分：能说出“先更新DB，再删缓存”的步骤，但对原理和优点解释不清。



3分：听说过这个方案，但描述不完整。



2分：提出错误的方案（如先删缓存再更新DB），且无法解释风险。
