CAP理论回顾：



一致性 (Consistency): 所有节点同时看到相同数据

方案B：TCC (Try-Confirm-Cancel)



java

@Tcc

public void createOrder(Order order) {

    // Try：预留资源

    stockService.tryLock(order.getSkuId(), order.getCount());

 

    // Confirm：确认扣减（Try成功后）

    stockService.confirm(order.getSkuId(), order.getCount());

 

    // Cancel：取消（Try失败时）

    stockService.cancel(order.getSkuId(), order.getCount());

}

// 1. Redis 预扣减（原子操作）

public boolean tryDecrementStock(String skuId, int count) {

    // Lua脚本保证原子性

    String lua = "local stock = redis.call('get', KEYS\[1]) " +

                 "if stock and tonumber(stock) >= tonumber(ARGV\[1]) then " +

                 "redis.call('decrby', KEYS\[1], ARGV\[1]) return 1 " +

                 "else return 0 end";

 

    Long result = (Long) redisTemplate.execute(

        new DefaultRedisScript<>(lua, Long.class),

        Arrays.asList("stock:" + skuId),

        String.valueOf(count)

    );

    return result == 1L;

}

@RabbitListener(queues = "stock-update")

-- 找到慢查询并Kill

SHOW FULL PROCESSLIST;

KILL <thread\_id>;



-- 临时停止从库查询（紧急恢复）

STOP SLAVE;

-- 等待从库恢复正常

START SLAVE;

预防措施：



从库设置read\_only



使用pt-kill自动Kill慢查询



增加从库数量，分担查询压力



场景3：网络问题导致



现象：



Slave\_IO\_Running: Connecting 或 No



Last\_IO\_Error 有网络相关错误



解决方案：



bash

\# 检查网络连通性

ping <master-ip>

telnet <master-ip> 3306



\# 重启复制线程

STOP SLAVE;

START SLAVE;



\# 如果网络不稳定，增加超时时间

STOP SLAVE;

CHANGE MASTER TO

    MASTER\_CONNECT\_RETRY=10,

    MASTER\_RETRY\_COUNT=86400;

START SLAVE;

场景4：数据冲突导致复制中断



现象：

解决方案：



sql

-- 跳过指定错误（根据错误码）

START SLAVE;



-- 或者设置跳过错误类型

STOP SLAVE;

SET GLOBAL slave\_skip\_errors = 1062;  -- 跳过主键冲突

START SLAVE;
