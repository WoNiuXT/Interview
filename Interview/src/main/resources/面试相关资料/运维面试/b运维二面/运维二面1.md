Q1：Linux 系统中，进程的几种状态分别是什么？特别是 R、S、D、Z 状态各代表什么含义？什么情况下会出现大量 D 状态进程？如何排查？

考察点： 对进程状态的理解，以及异常状态排查能力



答案要点 (来自《操作系统面试题》、《Linux系统面试题》):



进程状态含义：



R (TASK\_RUNNING): 正在运行或在运行队列中等待运行



S (TASK\_INTERRUPTIBLE): 可中断睡眠，等待某个条件（如I/O完成、信号），可被信号唤醒



D (TASK\_UNINTERRUPTIBLE): 不可中断睡眠，通常是在等待I/O（如磁盘、网络），不响应信号，不能被动杀死



Z (TASK\_ZOMBIE): 僵尸状态，进程已终止但父进程尚未调用wait()回收其PCB



T (TASK\_STOPPED): 暂停状态，通常是被信号暂停（如SIGSTOP）



t (TASK\_TRACED): 被调试器跟踪



大量 D 状态进程的原因：



磁盘I/O过载或故障（磁盘损坏、控制器问题）



NFS 服务端无响应（网络问题或服务端挂掉）



存储设备响应慢（如 SAN/NAS 延迟高）



内存压力过大，导致频繁换页



文件系统锁竞争或死锁



排查方法：



使用 top 或 ps -eo state,pid,cmd | grep "^D" 确认 D 状态进程

iostat -x 1 查看磁盘 %util 和 await，判断磁盘是否瓶颈

iotop 查看具体进程I/O情况

dmesg | tail 检查内核日志，看是否有I/O错误或磁盘超时

如果是 NFS，检查 NFS 服务端状态和网络连通性

strace -p <PID> 跟踪进程，看卡在哪个系统调用

如果无法恢复，可能需要重启服务或系统（作为最后手段）

预防措施：



使用高性能存储，监控磁盘健康状态



设置合理的 I/O 调度器（如 deadline, noop）



NFS 使用硬挂载时设置 intr 选项，允许中断



配置合理的文件系统参数



使用内存充足，避免频繁换页



Q2：Linux 的内存管理机制中，什么是 Page Cache？什么是 Buffer Cache？它们有什么区别？如果系统内存不足，应该先清理哪个？怎么清理？

考察点： 对 Linux 内存管理深度的理解



答案要点 (来自《操作系统面试题》):



Page Cache (页缓存):



用于缓存文件系统的文件内容（读/写文件）



当读取文件时，数据先进入 Page Cache，下次读直接命中，加速文件访问



写文件时，数据先写入 Page Cache（脏页），后台线程异步刷回磁盘



对应 free 命令中的 cached 列



Buffer Cache (缓冲区缓存):



用于缓存块设备（如磁盘分区）的元数据，如 superblock、inode



也缓存原始磁盘块（绕过文件系统直接读写块设备）



在现代 Linux 内核中，Buffer Cache 和 Page Cache 已部分整合，Buffer Cache 是 Page Cache 的一部分



对应 free 命令中的 buffers 列



区别：



Page Cache 以页为单位（通常 4KB），缓存文件内容



Buffer Cache 以块为单位，缓存设备元数据和块设备 I/O



Page Cache 面向文件系统，Buffer Cache 面向块设备



两者协同工作：Buffer Cache 记录 Page Cache 中哪些块是脏的



内存不足时的处理：



先清理 Page Cache（因为文件数据可以从磁盘重新读）



Buffer Cache 通常较小，但涉及文件系统元数据，清理风险较高



清理命令：



bash

\# 清理 Page Cache (文件缓存)

echo 1 > /proc/sys/vm/drop\_caches



\# 清理 dentries 和 inodes (目录项和索引节点)

echo 2 > /proc/sys/vm/drop\_caches



\# 清理 Page Cache、dentries 和 inodes (谨慎使用)

echo 3 > /proc/sys/vm/drop\_caches

注意事项：



清理缓存应谨慎，会影响系统性能（重新读数据）



生产环境建议优先通过增加内存解决，而非依赖手动清理



监控 sar -r 查看内存使用趋势



可以设置 vm.vfs\_cache\_pressure 控制内核回收缓存倾向



Q3：Linux 的 I/O 调度器有几种？各自的适用场景是什么？如何查看和修改当前使用的 I/O 调度器？

考察点： 对存储 I/O 调优的理解



答案要点 (来自《Linux系统面试题》):



I/O 调度器类型：



1\. CFQ (Completely Fair Queueing) - 完全公平队列



为每个进程分配时间片，公平分配 I/O 带宽



适合传统机械硬盘（HDD），多任务环境



能较好地隔离进程间的 I/O 影响



吞吐量较高，但延迟可能不稳定



2\. Deadline (截止时间调度器)



为每个 I/O 请求设置截止时间，防止请求饿死



读请求优先级高于写请求（读操作通常同步，影响应用）



适合数据库等需要保证响应时间的场景



机械硬盘和 SSD 均可使用



3\. NOOP (No Operation)



最简单的调度器，将 I/O 请求直接合并后放入队列



不做排序，依赖底层设备自身调度（如 SSD、RAID卡）



适合 SSD、NVMe、虚拟化环境（KVM、Xen）



减少 CPU 开销，延迟最低



4\. Anticipatory (预期调度器，已淘汰)



预测进程可能会继续读，延迟处理，已基本被替代



5\. MQ-DEADLINE (多队列 Deadline)



现代多队列块层的 Deadline 实现



支持多核并发，适合 NVMe SSD



6\. Kyber (轻量级调度器)



为高速设备设计，自动调节，开销小



适用场景总结：



机械硬盘、数据库：Deadline



传统服务器多任务：CFQ



SSD、NVMe、虚拟化：NOOP 或 MQ-DEADLINE



云环境虚拟磁盘：通常建议 NOOP



查看和修改命令：



bash

\# 查看当前 I/O 调度器

cat /sys/block/sda/queue/scheduler



\# 临时修改 I/O 调度器

echo deadline > /sys/block/sda/queue/scheduler



\# 永久修改（grub 添加内核参数）

\# 编辑 /etc/default/grub，添加 elevator=deadline

\# GRUB\_CMDLINE\_LINUX="... elevator=deadline"

\# update-grub 或 grub2-mkconfig

Q4：TCP 的拥塞控制算法有哪些？慢启动、拥塞避免、快速重传、快速恢复分别是什么？

考察点： 对 TCP 拥塞控制机制的深入理解



答案要点 (来自《计算机网络面试题》):



TCP 拥塞控制的四个核心算法：



1\. 慢启动 (Slow Start)



连接建立或超时后，拥塞窗口 cwnd 从 1 开始指数增长



每收到一个 ACK，cwnd 增加 1（实际上是每个 RTT 翻倍）



目的：快速探测网络可用带宽



直到达到慢启动阈值 ssthresh，进入拥塞避免



2\. 拥塞避免 (Congestion Avoidance)



cwnd 超过 ssthresh 后，进入线性增长阶段



每个 RTT 内，cwnd 增加 1（加法增加）



目的：缓慢探测网络容量，避免突然拥塞



3\. 快速重传 (Fast Retransmit)



接收方收到乱序包，会立即发送重复 ACK



发送方收到 3 个重复 ACK，认为报文丢失，不等超时就立即重传



提高丢包恢复速度，避免等待 RTO 超时



4\. 快速恢复 (Fast Recovery)



执行快速重传后，不回到慢启动，而是进入快速恢复



将 ssthresh 设为 cwnd/2，cwnd 设为 ssthresh + 3



继续发送新数据，收到新 ACK 后，进入拥塞避免



目的：避免网络突然变空，提高带宽利用率



拥塞控制算法演进：



Reno (经典)



实现上述四个基本算法



但一个 RTT 内只能恢复一个丢包



NewReno (改进)



改进快速恢复，一个 RTT 内可恢复多个丢包



Cubic (Linux 默认)



使用三次函数增长窗口，在高带宽高延迟网络中表现更好



RTT 无关的窗口增长，适合长肥网络



BBR (Bottleneck Bandwidth and RTT)



Google 开发的基于模型的新算法



不基于丢包，而是测量带宽和 RTT



适合高丢包网络，如无线网络



查看当前拥塞控制算法：



bash

sysctl net.ipv4.tcp\_congestion\_control

Q5：TIME\_WAIT 状态的作用是什么？过多的 TIME\_WAIT 连接会有什么问题？如何优化？

考察点： TCP 连接管理深度的理解，以及高并发场景优化经验



答案要点 (来自《计算机网络面试题》):



TIME\_WAIT 状态的作用：



出现在主动关闭连接的一方（客户端或服务端）



持续 2MSL（Maximum Segment Lifetime，通常 2分钟）



作用1： 确保最后一个 ACK 能被对方收到。如果 ACK 丢失，对方会重传 FIN，TIME\_WAIT 状态的连接可以重发 ACK



作用2： 让旧连接的延迟报文在网络中消失，避免干扰新连接（相同四元组）



过多 TIME\_WAIT 的问题：



占用大量内存（每个连接几 KB）



占用本地端口（客户端端口耗尽，无法建立新连接）



可能导致服务端性能下降，连接建立变慢



尤其在短连接高并发场景（如 Web 服务）更明显



优化方案：



1\. 开启端口复用 (安全)



bash

\# 允许 TIME\_WAIT 状态的端口被重用

sysctl -w net.ipv4.tcp\_tw\_reuse=1

仅适用于客户端（发起连接方）



需要配合 tcp\_timestamps 使用



2\. 加快回收 (谨慎使用，较新内核已移除)



bash

\# 快速回收 TIME\_WAIT 连接（Linux 4.12+ 已移除）

sysctl -w net.ipv4.tcp\_tw\_recycle=1

注意：NAT 环境下会导致问题（不同客户端时间戳冲突）



现代内核已废弃此参数



3\. 调整 MSL 时间



bash

\# 减小 MSL 时间（不推荐，可能影响可靠性）

\# 需要重新编译内核或修改内核参数

4\. 长连接替代短连接



使用 HTTP Keep-Alive 复用连接



减少连接建立和关闭次数



5\. 调高本地端口范围



bash

sysctl -w net.ipv4.ip\_local\_port\_range="1024 65535"

6\. 负载均衡层优化



如果服务端是 Nginx/Tomcat，可考虑让客户端主动关闭连接



Nginx 配置 keepalive\_requests 和 keepalive\_timeout



查看 TIME\_WAIT 数量：



bash

ss -tan state time-wait | wc -l

netstat -tan | grep TIME\_WAIT | wc -l

Q6：HTTPS 的完整握手过程是怎样的？TLS 1.2 和 TLS 1.3 的主要区别是什么？

考察点： 对现代网络协议加密机制的理解



答案要点 (来自《计算机网络面试题》):



TLS 1.2 完整握手过程 (4次往返):



1\. Client Hello



客户端发送支持的 TLS 版本、加密套件列表、压缩方法、随机数 ClientRandom



2\. Server Hello



服务器选择加密套件和 TLS 版本，发送服务器随机数 ServerRandom



3\. Certificate (服务器证书)



服务器发送数字证书（包含公钥、颁发者、有效期等）



4\. Server Key Exchange (可选)



如果选择的加密套件需要，服务器发送额外参数（如 DH 参数）



5\. Certificate Request (可选)



如果需要客户端认证，服务器请求客户端证书



6\. Server Hello Done



服务器握手完成



7\. Client Certificate (可选)



客户端发送证书



8\. Client Key Exchange



客户端生成预主密钥 PreMasterSecret，用服务器公钥加密后发送



双方用 ClientRandom、ServerRandom、PreMasterSecret 计算会话密钥



9\. Certificate Verify (可选)



客户端证明自己拥有私钥



10\. Change Cipher Spec



客户端通知后续通信将加密



11\. Finished



客户端发送加密的握手消息摘要，供服务器验证



12\. Change Cipher Spec



服务器通知后续通信将加密



13\. Finished



服务器发送加密的握手消息摘要，供客户端验证



TLS 1.3 主要改进 (1.5次往返):



1\. 简化握手过程



将 Client Hello 和 Key Exchange 合并



客户端直接发送支持的加密套件和自己的密钥共享参数



服务器回复 Server Hello + Certificate + Finished



1-RTT 完成握手（首次连接）



2\. 0-RTT 恢复



之前连接过的客户端可以发送 0-RTT 数据



直接发送加密的应用数据，同时进行握手



适用幂等请求（如 HTTP GET）



3\. 移除不安全特性



移除静态 RSA 密钥交换（不支持前向安全）



移除压缩、重协商等不安全功能



移除 MD5、SHA-1 等弱哈希算法



4\. 强制前向安全



所有密钥交换都必须支持 PFS（Perfect Forward Secrecy）



即使服务器私钥泄露，也无法解密之前的通信



5\. 简化加密套件



只支持 AEAD 认证加密（如 AES-GCM、ChaCha20-Poly1305）



加密和认证合一，减少出错可能



6\. 减少握手消息



从 13 条消息减少到 3-4 条消息



显著降低延迟



性能对比：



TLS 1.2：完整握手需 2 RTT



TLS 1.3：完整握手仅需 1 RTT，恢复会话可 0 RTT

Q7：负载均衡的常见算法有哪些？各自的优缺点和适用场景是什么？

考察点： 对负载均衡原理的理解，以及根据场景选型能力



答案要点 (来自《IT运维面试题》、《计算机网络面试题》):



1\. 轮询 (Round Robin)



原理： 依次将请求分发到后端服务器，循环往复



优点： 简单，无状态，实现容易



缺点： 不考虑服务器性能和当前负载，可能导致负载不均



适用场景： 后端服务器配置相同，请求处理时间相近的场景



2\. 加权轮询 (Weighted Round Robin)



原理： 根据服务器权重分配请求，权重高的获得更多请求



优点： 支持异构服务器，可灵活调整



缺点： 需要人工维护权重，可能不准确



适用场景： 服务器性能不一，需要根据实际能力分配流量



3\. 最少连接 (Least Connections)



原理： 将请求分发到当前活跃连接数最少的服务器



优点： 动态感知服务器负载，适合长连接场景



缺点： 需要维护连接状态，实现复杂



适用场景： 长连接服务（如 WebSocket）、请求处理时间不均的场景



4\. 源地址哈希 (Source IP Hash)



原理： 根据客户端 IP 计算哈希值，固定分配到某台服务器



优点： 保持会话，适合需要会话粘性的应用



缺点： 服务器增减会导致大量重哈希，可能负载不均



适用场景： 需要会话保持且不用共享 Session 的场景



5\. 一致性哈希 (Consistent Hash)



原理： 改进的哈希算法，节点增减只影响少量映射关系



优点： 平滑扩展，适合缓存等分布式系统



缺点： 实现复杂，需要虚拟节点处理负载不均



适用场景： 分布式缓存（如 Memcached、Redis Cluster）



6\. 最快响应 (Fastest Response)



原理： 根据服务器响应时间动态分配，响应快的获得更多请求



优点： 自适应性能变化，用户体验好



缺点： 需要实时监测，可能受网络波动影响



适用场景： 全局负载均衡（GSLB），跨地域部署



7\. 随机 (Random)



原理： 随机选择一台服务器



优点： 简单，在大并发下分布均匀



缺点： 单次请求可能分布不均



适用场景： 服务器数量多，请求量大的场景



Nginx 配置示例：



nginx

upstream backend {

&nbsp;   # 默认轮询

&nbsp;   server 192.168.1.10 weight=3;  # 加权轮询

&nbsp;   server 192.168.1.11;

&nbsp;   

&nbsp;   # 最少连接

&nbsp;   least\_conn;

&nbsp;   

&nbsp;   # IP哈希

&nbsp;   ip\_hash;

&nbsp;   

&nbsp;   # 一致性哈希（需第三方模块）

&nbsp;   hash $request\_uri consistent;

}

Q8：LVS、Nginx、HAProxy 这三种负载均衡器有什么区别？各自适用什么场景？

考察点： 对负载均衡技术栈的全面掌握



答案要点 (来自《IT运维面试题》):



LVS (Linux Virtual Server) - 传输层 (Layer 4)



特点：



工作在 OSI 模型第 4 层（传输层）



基于 IP 和端口进行转发，不解析应用层协议



性能极高，可支撑百万级并发



支持三种模式：DR (Direct Routing)、NAT、TUN (Tunneling)



支持多种调度算法：轮询、加权轮询、最少连接、源地址哈希等



优点：



性能最好，抗负载能力强



稳定性高，工作在内核空间



配置简单，功能稳定



缺点：



不支持七层功能（如 URL 路由、SSL 卸载）



无法基于内容做分发



健康检查功能较弱



适用场景：



大型网站入口层，处理海量流量



需要极高吞吐量的场景



作为最前端的流量入口



Nginx - 应用层 (Layer 7)



特点：



工作在 OSI 模型第 7 层（应用层）



基于 HTTP/HTTPS 协议内容进行分发



支持 URL 路由、正则匹配、重写规则



支持 SSL 终止、缓存、压缩等功能



高并发能力也较强（基于 epoll）



优点：



功能丰富，支持七层处理



配置灵活，模块化设计



可作为 Web 服务器、反向代理、缓存服务器



支持健康检查和动态负载均衡



缺点：



性能低于 LVS（需要解析应用层）



配置重载需要信号，不是实时生效



适用场景：



Web 应用的反向代理



动静分离，静态资源服务



SSL 终端卸载



API 网关



HAProxy - 混合层 (Layer 4/7)



特点：



支持四层和七层负载均衡



专为高可用、负载均衡设计的专业软件



支持 TCP 和 HTTP 协议



拥有丰富的统计信息和管理接口



健康检查非常强大（多种检查方式）



优点：



四层性能接近 LVS，七层功能丰富



支持会话保持（多种方式）



提供详细的状态监控页面



配置语法清晰，易于维护



缺点：



不能作为 Web 服务器（无静态文件服务）



配置变更通常需要重启



适用场景：



TCP 服务的负载均衡（如 MySQL、Redis）



Web 服务的负载均衡（替代 Nginx）



需要详细监控和统计的场景



选型对比：



特性	LVS	Nginx	HAProxy

工作层	四层	七层	四/七层

Q9：Kubernetes 的核心组件有哪些？它们各自的作用是什么？

考察点： 对容器编排平台架构的理解



答案要点 (来自《DevOps运维面试题》):



控制平面 (Control Plane) 组件：



1\. kube-apiserver (API 服务器)



集群的入口，所有操作都通过 API Server



提供 REST API 接口，用于组件间通信



负责认证、授权、准入控制



唯一与 etcd 直接交互的组件



2\. etcd (分布式键值存储)



存储集群的所有配置信息和状态数据



使用 Raft 协议保证一致性



需要定期备份，通常部署奇数个节点（3/5/7）



3\. kube-scheduler (调度器)



监听新创建的 Pod，选择合适的 Node 运行



考虑资源需求、亲和性/反亲和性、污点容忍等约束



使用过滤和打分两个阶段选择最佳节点



4\. kube-controller-manager (控制器管理器)



运行各种控制器，维持集群期望状态



包含：Node Controller、Replication Controller、Deployment Controller、Service Controller 等



通过 API Server 监控资源变化，并做出相应调整



5\. cloud-controller-manager (云控制器管理器) - 可选



与云厂商 API 交互



管理云资源（负载均衡、存储卷、节点）



工作节点 (Node) 组件：



6\. kubelet



每个节点上的代理，负责管理 Pod 生命周期



与容器运行时交互（如 containerd、CRI-O）



上报节点和 Pod 状态给 API Server



执行健康检查（liveness、readiness）



7\. kube-proxy



维护节点上的网络规则，实现 Service 的负载均衡



支持 iptables、IPVS、userspace 等模式



将访问 Service 的流量转发到后端 Pod



8\. 容器运行时 (Container Runtime)



负责拉取镜像、运行容器



支持 containerd、CRI-O、Docker（通过 cri-dockerd）



附加组件：



9\. CoreDNS



集群 DNS 服务，提供服务发现



Pod 可以通过 Service 名访问其他服务



10\. Ingress Controller



管理 Ingress 资源，提供 HTTP/HTTPS 路由



常见实现：Nginx Ingress Controller、Traefik



11\. Network Plugin (CNI)



实现 Pod 网络通信



如 Calico、Flannel、Weave、Cilium



工作流程示例：



text

用户 (kubectl) → API Server → etcd(存储) → Scheduler(调度) → API Server → kubelet(执行) → 容器运行时 → Pod 运行

Deployment (无状态应用)

管理无状态应用，所有 Pod 可互换



Pod 名称随机，无持久身份



支持滚动更新、回滚、扩缩容



支持声明式更新策略

适用场景：



Web 应用、API 服务、微服务

Q11：Kubernetes 中，Service 有哪些类型？ClusterIP、NodePort、LoadBalancer 的区别是什么？

考察点： 对服务暴露方式的理解



答案要点 (来自《DevOps运维面试题》):



Service 类型：



1\. ClusterIP (默认类型)



在集群内部提供一个虚拟 IP（Cluster IP）



仅集群内可访问，外部无法直接访问



适用于内部服务调用（如前端调用后端）



原理：



kube-proxy 在节点上创建 iptables/IPVS 规则



将访问 ClusterIP 的流量转发到后端 Pod



示例：



yaml

apiVersion: v1

kind: Service

metadata:

&nbsp; name: my-service

spec:

&nbsp; type: ClusterIP

&nbsp; selector:

&nbsp;   app: my-app

&nbsp; ports:

&nbsp; - protocol: TCP

&nbsp;   port: 80

&nbsp;   targetPort: 8080

2\. NodePort



在每个节点上开放一个静态端口（30000-32767）



通过 NodeIP:NodePort 可从外部访问



自动创建 ClusterIP（流量路径：NodePort → ClusterIP → Pod）



优点：



简单，无需云厂商支持



可用于本地测试或小规模暴露



缺点：



端口范围有限（默认30000-32767）



需要知道节点 IP（节点故障需重新路由）



安全性较低（直接暴露节点端口）



示例：



yaml

apiVersion: v1

kind: Service

metadata:

&nbsp; name: my-service

spec:

&nbsp; type: NodePort

&nbsp; selector:

&nbsp;   app: my-app

&nbsp; ports:

&nbsp; - port: 80

&nbsp;   targetPort: 8080

&nbsp;   nodePort: 30080  # 可选，不指定则随机分配

3\. LoadBalancer



在 NodePort 基础上，调用云厂商 API 创建外部负载均衡器



将外部请求分发到各节点的 NodePort



适用于公有云环境（AWS ELB、阿里云 SLB）



优点：



提供公网访问入口



云厂商负载均衡器高可用



可配合 DNS 实现域名访问



缺点：



依赖云厂商



每个 Service 创建单独的 LB，成本高



不适合大规模服务暴露



示例：



yaml

apiVersion: v1

kind: Service

metadata:

&nbsp; name: my-service

spec:

&nbsp; type: LoadBalancer

&nbsp; selector:

&nbsp;   app: my-app

&nbsp; ports:

&nbsp; - protocol: TCP

&nbsp;   port: 80

&nbsp;   targetPort: 8080

4\. ExternalName



特殊类型，将 Service 映射到外部域名



返回 CNAME 记录，不创建 ClusterIP



适用于访问集群外服务（如数据库）



示例：



yaml

apiVersion: v1

kind: Service

metadata:

&nbsp; name: external-db

spec:

&nbsp; type: ExternalName

&nbsp; externalName: database.example.com

流量路径对比：



text

ClusterIP:

客户端(Pod) → ClusterIP:80 → iptables → Pod:8080



NodePort:

外部客户端 → NodeIP:30080 → iptables → ClusterIP → Pod:8080



LoadBalancer:

外部客户端 → LB IP:80 → NodeIP:30080 → iptables → ClusterIP → Pod:8080

实际应用架构：



内部服务调用：ClusterIP



测试环境暴露：NodePort



生产环境对外服务：LoadBalancer + Ingress



Q12：Kubernetes 中，如何进行滚动更新和回滚？如何控制更新速度？

考察点： 对应用发布策略的理解



答案要点 (来自《DevOps运维面试题》):



滚动更新 (Rolling Update)：



原理：



逐步替换旧版本 Pod，保持服务不中断



Deployment 创建新的 ReplicaSet，逐步增加新 Pod



同时减少旧 ReplicaSet 的 Pod 数量



更新过程中新旧版本共存



触发滚动更新：



bash

\# 更新镜像版本

kubectl set image deployment/nginx-deployment nginx=nginx:1.25



\# 更新配置（如环境变量、资源限制）

kubectl edit deployment/nginx-deployment



\# 直接应用新 YAML

kubectl apply -f deployment.yaml

查看更新状态：



bash

\# 查看更新状态

kubectl rollout status deployment/nginx-deployment



\# 查看历史版本

kubectl rollout history deployment/nginx-deployment



\# 查看详情

kubectl rollout history deployment/nginx-deployment --revision=2

回滚操作：



bash

\# 回滚到上一个版本

kubectl rollout undo deployment/nginx-deployment



\# 回滚到指定版本

kubectl rollout undo deployment/nginx-deployment --to-revision=1



\# 暂停和恢复

kubectl rollout pause deployment/nginx-deployment

kubectl rollout resume deployment/nginx-deployment

控制更新速度的关键参数：



1\. maxSurge



滚动更新时，允许超过期望副本数的最大 Pod 数



可以是绝对数值或百分比



默认 25%，表示可额外创建 25% 的 Pod



2\. maxUnavailable



更新过程中，允许不可用的最大 Pod 数



可以是绝对数值或百分比



默认 25%，表示可容忍 25% 的 Pod 不可用



3\. minReadySeconds



Pod 进入 Ready 状态后，等待指定时间才继续更新



用于给应用预热时间



默认 0



4\. progressDeadlineSeconds



如果滚动更新在此时间内未完成，标记为失败



默认 600 秒



配置示例：



yaml

apiVersion: apps/v1

kind: Deployment

metadata:

&nbsp; name: nginx-deployment

spec:

&nbsp; replicas: 10

&nbsp; strategy:

&nbsp;   type: RollingUpdate

&nbsp;   rollingUpdate:

&nbsp;     maxSurge: 2          # 最多允许额外2个Pod

&nbsp;     maxUnavailable: 1    # 最多允许1个Pod不可用

&nbsp; minReadySeconds: 30      # Pod Ready后等待30秒

&nbsp; progressDeadlineSeconds: 600

&nbsp; selector:

&nbsp;   matchLabels:

&nbsp;     app: nginx

&nbsp; template:

&nbsp;   ...

更新策略选择：



1\. 快速更新



yaml

rollingUpdate:

&nbsp; maxSurge: 50%

&nbsp; maxUnavailable: 50%

更新速度快



可能影响容量（一半 Pod 同时不可用）



2\. 保守更新



yaml

rollingUpdate:

&nbsp; maxSurge: 1

&nbsp; maxUnavailable: 0

保证容量（先创建新 Pod，再删除旧 Pod）



更新速度慢，需要额外资源



3\. 蓝绿部署 (通过修改 Selector)



先创建新版本（绿色）



测试通过后，修改 Service Selector 指向新版本



然后删除旧版本（蓝色）



回滚原理：



Deployment 保留历史 ReplicaSet（默认保留 10 个）



回滚时，将目标 ReplicaSet 的 replicas 数调整到期望值



当前 ReplicaSet 的 replicas 数调整到 0



实现了快速回滚

Q14：什么是可观测性？Metrics、Logs、Traces 三者的关系是什么？如何选择工具？

考察点： 对现代运维体系的整体认知



答案要点 (来自《IT运维面试题》):



可观测性 (Observability) 定义：



通过系统的外部输出（指标、日志、链路）来推断内部状态的能力



不仅监控“已知的未知”，还能发现“未知的未知”



三支柱：Metrics（指标）、Logs（日志）、Traces（链路）



三支柱详解：



1\. Metrics (指标) - 聚合的、数值化的数据



特点：



时间序列数据，存储开销小



支持聚合、降采样、长期存储



适合趋势分析和告警



问题：只能看到“有问题”，看不到“具体原因”



典型指标类型：



Counter（累计值，如请求总数）



Gauge（当前值，如内存使用率）



Histogram/Summary（分布统计，如响应延迟）



工具： Prometheus、Graphite、InfluxDB、CloudWatch



2\. Logs (日志) - 离散的、结构化的事件



特点：



最丰富的信息来源，包含上下文



支持全文搜索和过滤



适合问题排查和审计



问题：数据量大，存储成本高，难以聚合统计



日志类型：



应用日志（业务日志、错误日志）



系统日志（syslog、内核日志）



访问日志（Nginx/Apache 日志）



工具： ELK/EFK (Elasticsearch + Filebeat + Kibana)、Loki、Splunk



3\. Traces (链路追踪) - 请求的全貌



特点：



追踪一个请求经过的所有服务



展示调用关系、耗时分布



适合性能瓶颈分析和依赖梳理



问题：采样率低，实现复杂



核心概念：



Trace（完整请求链路）



Span（单个服务的处理单元）



Trace ID（全局唯一标识）



工具： Jaeger、Zipkin、SkyWalking、Pinpoint



三者的关系：



text

一个请求 → 生成 Trace（完整路径）

&nbsp;      ↓

各服务记录 Logs（具体细节）

&nbsp;      ↓

聚合生成 Metrics（统计趋势）

协同工作流程：



告警触发： Metrics 检测到错误率上升

初步定位： 查看错误率高的服务和接口

链路分析： 通过 Trace 找到具体哪个环节慢或失败

日志排查： 查看该 Trace 对应的详细日志，定位代码级问题

工具选型对比：



领域	开源方案	商业方案	云厂商方案

Metrics	Prometheus + Grafana	Datadog	ARMS、CloudWatch

Logs	ELK/EFK、Loki	Splunk	SLS、Cloud Logging

Traces	Jaeger、SkyWalking	New Relic	XTrace、Cloud Trace

一体化工具有：



Grafana (支持 Prometheus、Loki、Tempo)



Elastic APM (整合日志和链路)



SkyWalking (原生支持三者)



选型考量因素：



数据规模： 每天产生多少日志/指标



查询需求： 是否需要全文搜索



预算： 商业版成本 vs 自建运维成本



团队技能： 熟悉哪些技术栈



云环境： 使用云厂商托管服务 vs 自建

Q17：你如何看待 DevOps 文化？在你的团队中，你是如何推动开发与运维协作的？

考察点： 对 DevOps 理念的理解和实践经验



答案要点 (综合文档):



DevOps 的理解：



DevOps 不是工具，也不是职位，而是一种文化和实践



核心目标：打破开发和运维之间的壁垒，提升交付效率



三个关键：文化、自动化、度量



DevOps 核心原则：



文化： 共享责任、互相信任、共同目标

自动化： CI/CD、基础设施即代码、自动化测试

度量： 用数据说话，持续改进

共享： 知识共享、责任共担、工具共享

推动开发运维协作的实践：



1\. 建立共同目标



不考核“上线次数”和“故障次数”，而是考核“系统可用性”



开发关注“功能交付”，运维关注“稳定性”，共同目标是“用户满意”



2\. 共享 On-Call 轮值



开发人员参与线上 On-Call



让开发直接感受线上故障的影响



促进开发写出更健壮的代码



3\. 统一工具链



统一 CI/CD 平台（Jenkins/GitLab CI）



统一监控告警平台（Prometheus + Grafana）



统一日志平台（ELK）



减少沟通成本，提高效率



4\. 基础设施即代码



使用 Terraform 管理云资源



使用 Ansible 配置服务器



所有变更通过代码 review，避免“运维黑盒”



5\. 建立 SLO 和服务等级



yaml

\# 示例 SLO

service: order-api

slos:

&nbsp; - name: availability

&nbsp;   target: 99.99%

&nbsp;   window: 30d

&nbsp; - name: latency

&nbsp;   target: 100ms (P99)

&nbsp;   window: 30d

6\. 定期复盘和分享



每月一次“故障复盘会”，不追责，只找改进点



开发分享新功能设计，运维分享线上问题



共同维护知识库，沉淀经验



7\. 共建运维平台



开发自服务：自助创建测试环境、自助发布



运维提供平台和工具，而不是替开发操作



例如：开发通过 GitLab CI 触发部署，无需找运维



效果度量：



部署频率（从每月到每周/每天）



变更前置时间（从几天到几小时）



故障恢复时间（MTTR）



变更失败率
