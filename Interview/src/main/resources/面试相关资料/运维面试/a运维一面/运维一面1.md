1.1 进程与资源管理

Q1：请详细解释 Linux 系统中进程和线程的区别。在实际的应用场景中，比如 Nginx 和 Tomcat，它们分别是如何利用进程/线程模型的？



考察点： 操作系统核心概念的理解，并能联系实际应用。



答案要点 (来自《操作系统面试题》、《IT运维面试题》):



核心区别：



资源分配单位 vs 调度单位： 进程是资源分配的最小单位（拥有独立的内存空间、文件描述符等）；线程是CPU调度的最小单位（共享进程的资源）。

隔离性： 进程间相互隔离，一个进程崩溃通常不影响其他进程；线程共享进程内存，一个线程出错可能导致整个进程退出。

通信与开销： 进程间通信复杂（管道、共享内存、Socket等），创建开销大；线程间通信简单（直接读写共享变量），创建开销小。

应用场景举例：



Nginx： 采用多进程模型（master-worker）。Master进程管理，Worker进程处理请求。这样设计是为了高稳定性和利用CPU多核，每个Worker进程独立，避免内存泄漏相互影响。



Tomcat： 采用多线程模型。一个Java进程内启动多个线程来处理并发的HTTP请求，共享JVM堆内存和连接池，资源开销小，适合高并发短连接场景。



Q2：Linux 系统中，如何查看某个特定进程（比如PID为1234）的内存使用详情？有哪些不同粒度的查看方式？



考察点： 对进程内存监控工具的掌握程度，从粗到细。



答案要点 (来自《IT运维面试题》):



粗粒度： ps -p 1234 -o pid,rss,vsize,%mem,comm。重点关注 rss (实际物理内存)。

动态监控： top -p 1234 或 htop，可以实时观察内存变化。

细粒度分析： pmap -x 1234。可以列出进程每一块内存映射的详细信息，包括堆、栈、共享库的大小，非常适合排查内存泄漏。

命令执行峰值： /usr/bin/time -v command\_name，可以统计一个命令运行过程中的最大驻留集大小。

Q3：什么是 Linux 的 CFS 调度器？简单说说它的工作原理。



考察点： 对内核调度器的了解深度。



答案要点 (来自《操作系统面试题》):



CFS (Completely Fair Scheduler) 是 Linux 内核默认的进程调度器。



核心理念： 不为进程分配固定时间片，而是通过计算每个进程的虚拟运行时间 (vruntime) 来保证公平。



工作原理：



用红黑树管理所有可运行进程，键值是 vruntime。vruntime 值最小的进程，说明它过去占用的CPU时间最少，最应该被调度。

调度器每次选择红黑树最左边的节点（vruntime最小）来运行。

vruntime 的增长速度受进程优先级（nice值）影响，优先级高的进程 vruntime 增长慢，因此能获得更多CPU时间。

1.2 文件系统与存储

Q4：Linux 文件系统的基本目录结构是怎样的？请解释 /bin、/sbin、/etc、/var、/usr、/proc 这几个目录的主要作用。



考察点： FHS (Filesystem Hierarchy Standard) 标准掌握情况。



答案要点 (来自《Linux系统面试题》、《IT运维面试题》):



/bin：存放系统启动和运行所必需的基础命令（如 ls, cp, bash），普通用户也能执行。



/sbin：存放系统管理员使用的系统管理命令（如 fdisk, iptables, reboot），通常需要root权限。



/etc：配置文件的大本营，几乎所有服务的配置都存放在此。



/var：存放经常变化的数据，最典型的是日志文件 /var/log，还包括邮件队列、缓存等。



/usr：Unix System Resources，存放非必需的用户应用程序和文件，如 /usr/bin (更多用户命令)、/usr/lib (库文件)。



/proc：虚拟文件系统，不占磁盘空间，以文件形式提供内核和进程的运行时信息（如 /proc/cpuinfo, /proc/meminfo）。



Q5：硬链接和软链接的区别已经知道了。那如果我想修改一个文件的权限，这个操作会影响它的硬链接和软链接吗？为什么？



考察点： 对 inode 和文件属性的深入理解。



答案要点 (来自《Linux系统面试题》):



对硬链接： 会影响。因为硬链接与源文件共享同一个 inode，而文件的权限信息（如 rwx）是存储在 inode 中的。修改源文件的权限，本质上是修改了 inode 里的元数据，所以所有指向这个 inode 的硬链接的权限都会随之改变。



对软链接： 不影响软链接文件本身，但会影响通过软链接访问目标文件的结果。软链接有自己的独立 inode，存储的是目标文件的路径。修改源文件权限，是修改了目标文件的 inode。当你通过软链接访问时，系统会根据路径找到目标文件，并应用目标文件的新权限。软链接自身的权限（通常是 777）通常无关紧要。



Q6：磁盘空间报警，df -h 显示 / 根分区已用 100%，但 du -sh /\* 统计各目录大小加起来远小于总空间。可能是什么原因？如何解决？



考察点： 文件系统管理的实战经验，解决“空间未释放”问题。



答案要点 (来自《系统运维面试题》):



原因： 最常见的可能是某个大文件被删除了（rm -f），但还有进程在持续写入该文件，导致文件句柄未被释放。对于内核来说，这个文件仍被占用，所以空间不会真正释放给文件系统。



解决方法：



使用 lsof | grep deleted 命令查找所有被标记为 deleted 但仍有进程打开的文件。

找到占用文件的进程 PID。

根据业务情况，重启该进程 (kill -HUP <PID> 或 systemctl restart <service>)，或者直接 kill -9 <PID> (慎用)，让操作系统回收文件句柄，释放磁盘空间。

1.3 用户与权限

Q7：Linux 的权限管理机制中，除了常见的 rwx，还有哪些特殊权限？比如 Sticky Bit 和 SUID 有什么作用？



考察点： 对安全机制的深度了解。



答案要点 (来自《Linux系统面试题》):



Sticky Bit (粘滞位)： 通常用于共享目录（如 /tmp）。权限标志为 t。作用：即使目录对所有人可写，用户也只能删除属于自己的文件，不能删除其他人的文件。



SUID (Set User ID)： 设置在可执行文件上。当用户执行该程序时，进程的有效用户ID会临时切换为该程序文件所有者的ID。典型例子是 passwd 命令，普通用户执行时能临时获得 root 权限来修改 /etc/shadow 文件。



SGID (Set Group ID)： 设置在可执行文件或目录上。对于文件，效果类似SUID，但作用于组。对于目录，任何在该目录下创建的新文件或目录，都会自动继承该目录的所属组。

2.1 Shell 脚本进阶

Q9：Shell 脚本中，$@ 和 $\* 有什么区别？在 for 循环中遍历所有参数时，为什么通常推荐用 "$@"？



考察点： 对位置参数的深入理解，以及处理带空格参数的正确方式。



答案要点 (来自《系统运维面试题》):



区别： 两者都代表传递给脚本或函数的所有参数。



$\*：将所有参数视为一个字符串，参数之间用 $IFS（默认为空格）分隔。



$@：将每个参数视为一个独立的实体。



推荐 "$@" 的原因： 当参数本身包含空格时（例如 "hello world" 作为一个参数），使用 "$@" 可以正确地将其当作一个整体。而 "$\*" 会将 hello 和 world 拆成两个独立的元素。因此，在需要保持参数完整性的场景（如 for 循环、传递给其他命令），必须使用 "$@"。



Q10：写一个 Shell 函数，接收一个目录路径作为参数，检查该目录是否存在。如果不存在，则创建它（包括必要的父目录），并输出成功信息；如果存在，则输出目录已存在的提示。



考察点： 函数定义、条件判断、目录操作。

check\_and\_create\_dir() {

    local dir\_path="$1"

    if \[ -z "$dir\_path" ]; then

        echo "错误：未提供目录路径"

        return 1

    fi

    if \[ -d "$dir\_path" ]; then

        echo "目录 '$dir\_path' 已存在。"

    else

        # -p 参数可以创建多级父目录，且如果目录已存在不会报错

        mkdir -p "$dir\_path"

        if \[ $? -eq 0 ]; then

            echo "目录 '$dir\_path' 创建成功。"

        else

            echo "错误：无法创建目录 '$dir\_path'，请检查权限。"

            return 1

        fi

    fi

}

import subprocess



def get\_disk\_usage():

    try:

        # run() 方法是推荐的执行命令方式

        # capture\_output=True 捕获 stdout 和 stderr

        # text=True 将输出以文本形式返回（而非字节）

        # shell=False (默认) 使用列表形式传递命令更安全

        result = subprocess.run(\['df', '-h'], capture\_output=True, text=True, check=True)

        # 命令执行成功，result.stdout 即为输出内容

        print("磁盘使用情况：")

        print(result.stdout)

    except subprocess.CalledProcessError as e:

        # 命令执行失败，退出码非0

        print(f"命令执行失败，退出码：{e.returncode}")

        print(f"错误输出：{e.stderr}")

    except FileNotFoundError:

        print("错误：'df' 命令未找到，请检查系统。")

Q14：Python 中，\_\_name\_\_ == '\_\_main\_\_' 这行代码的作用是什么？为什么写脚本时通常要加上它？



考察点： 对 Python 模块导入机制的理解。



答案要点：



作用： 判断当前模块是被直接运行的，还是被导入到其他模块中。



原因： 当 .py 文件被直接运行时，\_\_name\_\_ 变量的值被设置为 '\_\_main\_\_'。如果文件是被 import 到另一个模块中，\_\_name\_\_ 会被设置为该模块的名字（即文件名）。



最佳实践： 加上 if \_\_name\_\_ == '\_\_main\_\_': 可以让你的脚本既可以作为独立的程序运行（执行测试代码或主逻辑），又可以被其他模块安全地导入，而不会自动执行那些测试代码。

3.1 TCP/IP 与 HTTP 基础

Q15：TCP 协议是如何保证可靠传输的？请列举几个关键机制。



考察点： 对 TCP 核心可靠机制的掌握。



答案要点 (来自《计算机网络面试题》):



序列号与确认应答 (ACK)： 每个字节都有序号，接收方通过 ACK 告知发送方已成功接收哪些数据。

超时重传： 发送方发送数据后启动定时器，若超时未收到 ACK，则重传该数据。

流量控制： 接收方通过通告窗口 (rwnd) 告知自己的接收能力，防止发送方发送过快导致缓冲区溢出。

拥塞控制： 发送方通过慢启动、拥塞避免、快重传、快恢复等算法动态调整发送速度，避免网络过载。

校验和： 检查数据在传输过程中是否损坏。

数据排序： 接收方根据序列号对收到的乱序数据包进行重新排序，保证数据顺序正确。

Q16：描述一下在浏览器输入 https://www.taobao.com 并回车后，发生了哪些网络层面的交互？(重点在DNS、TCP、TLS)



考察点： 对网络请求全流程的综合理解。



答案要点 (来自《计算机网络面试题》):



DNS 解析： 浏览器向本地DNS服务器发起递归查询，最终从淘宝的权威DNS服务器获取到域名对应的IP地址。

TCP 连接： 浏览器与目标IP的80（HTTP）或443（HTTPS）端口进行三次握手，建立TCP连接。

TLS 握手（HTTPS场景）：

Client Hello: 浏览器发送支持的TLS版本、加密套件列表。

Server Hello: 服务器选择加密套件，并发送数字证书。

浏览器验证证书合法性。

密钥交换：双方协商出会话密钥。

握手完成，开始加密通信。

HTTP 请求与响应： 浏览器发送HTTP GET请求，服务器处理后返回HTML内容。

TCP 连接关闭（或复用）： 如果是短连接，四次挥手关闭；如果是长连接，则可能复用。

Q17：HTTP 的 GET 和 POST 请求，从本质上（语义上）有什么区别？



考察点： 对 HTTP 方法语义的理解。



答案要点 (来自《计算机网络面试题》):



GET： 用于请求服务器的资源。它应该是安全的（不应引起服务器端任何状态变化）和幂等的（无论执行多少次，结果都一样）。



POST： 用于向服务器提交数据，请求服务器处理，例如创建一个新资源或提交表单。它不是安全的，通常也不是幂等的（多次提交可能会创建多个资源）。



本质区别不是参数位置或长度，而是语义。 GET的请求参数习惯上放在URL里，是因为它本身就是“获取”资源，参数是资源的标识符。POST的请求数据放在Body里，是因为它是“提交”的数据体。虽然技术上都可实现，但违背语义会导致缓存、代理等中间件行为异常。



3.2 网络故障排查命令

Q18：当用户反馈网站无法访问时，你在服务器上会用哪些命令来诊断问题？请从网络层到应用层逐层说明。



考察点： 系统化的网络故障排查思路。



答案要点 (来自《计算机网络面试题》、《IT运维面试题》):



链路层/网络层： ping <目标IP>。检查最基本的网络连通性。如果 ping 不通，问题在底层网络或防火墙。

路由追踪： traceroute <目标IP> (Linux) 或 tracert <目标IP> (Windows)。查看数据包经过的路由路径，定位在哪一跳丢包或延迟高。

端口与服务状态：

ss -tulnp | grep <端口号> 或 netstat -tulnp | grep <端口号>。检查服务是否在正确的端口上监听。

telnet <服务器IP> <端口号> 或 nc -v <服务器IP> <端口号>。从外部测试端口是否可达，检查防火墙策略。

应用层：

curl -I http://<服务器IP>:<端口>。直接发送HTTP请求，查看响应状态码，判断Web服务是否正常。

curl -v http://... 查看更多细节，如SSL握手情况。

DNS 检查： nslookup <域名> 或 dig <域名>。如果 ping IP 通，但 ping 域名 不通，肯定是DNS解析问题。

Q19：netstat 和 ss 命令都可以用来查看网络连接，它们有什么区别？生产环境更推荐用哪一个？为什么？



考察点： 对老旧工具和新工具的对比认知。



答案要点 (来自《Linux系统面试题》):



区别： netstat 是一个较老的、经典的网络工具。ss (socket statistics) 是它的现代替代品。



推荐 ss 的原因：



性能更高： 当服务器有大量连接（上万）时，netstat 需要遍历 /proc 文件系统，速度慢且消耗资源。ss 直接从内核空间获取信息，性能远超 netstat。

功能更强： ss 提供了更丰富、更灵活的过滤功能。

常用 ss 命令： ss -tulnp (查看所有监听端口)、ss -tn state established (查看已建立的TCP连接)、ss -tn sport = :80 (查看源端口为80的连接)。

Q25：Linux 系统中，top 命令输出的 load average (1分钟, 5分钟, 15分钟) 分别代表什么？如何根据这三个值判断系统负载趋势？

答案要点 (来自《操作系统面试题》):



含义： 分别代表过去1分钟、5分钟、15分钟内，处于可运行状态（R）和不可中断睡眠状态（D）的平均进程数。简单理解，就是“正在使用CPU”和“正在等待I/O”的进程平均数量。



趋势判断：



1分钟值 > 5分钟值 > 15分钟值： 表示负载正在上升，需要关注。



1分钟值 < 5分钟值 < 15分钟值： 表示负载正在下降，系统压力减轻。



三个值都非常高： 说明系统在过去15分钟内一直处于高负载状态，问题可能比较严重。



核心判断： 需要结合CPU核心数来看。一个4核CPU的机器，如果load average长期高于4，说明进程在排队等待CPU，系统存在瓶颈。



Q26：什么是 CPU 上下文切换？频繁的上下文切换会对系统性能产生什么影响？用什么命令可以查看？



考察点： 对操作系统调度成本的认知。



答案要点：



定义： 将 CPU 的控制权从一个进程或线程切换到另一个。这包括保存当前任务的上下文（寄存器、程序计数器等），并加载新任务的上下文。



影响： 上下文切换本身有开销（CPU时间）。频繁的切换会导致CPU把大量时间花费在“切换”这件事本身，而不是在真正执行任务上，导致系统吞吐量下降。

Q28：你的应用日志文件 app.log 增长非常快，你希望在不重启应用的情况下清空这个日志文件。正确的做法是什么？为什么不能直接 rm 或 > 覆盖？



考察点： 对 Linux 文件删除机制的深入理解，日志管理的经验。



答案要点 (来自《Linux系统面试题》):



正确做法： cat /dev/null > app.log 或 > app.log (使用 shell 重定向清空)。



为什么不能 rm？ 如果应用进程正在写 app.log，它会持有该文件的 inode。直接 rm 只是删除了文件名（目录项），但文件数据块和 inode 仍然被应用进程占用，磁盘空间不会立即释放，直到进程重启或关闭文件句柄。



为什么不能 > app.log？ 如果你直接 > app.log，它会截断文件。对于许多应用，特别是使用 fopen 以追加模式打开的文件，截断操作是安全的，并且会立即释放磁盘空间，不需要重启应用。这是最常用的方法。如果应用写入机制复杂，可能需要配合 logrotate 等工具。



5.3 故障处理场景题

Q29：线上服务器 CPU 使用率突然飙高到 100%，你 top 后发现是一个叫 java 的进程导致的。作为一名运维，你接下来的排查步骤是什么？



考察点： 综合故障排查思路。



答案要点：



确认进程： 记录下这个 Java 进程的 PID (例如 1234)。

定位高负载线程： 使用 top -H -p 1234，查看这个 Java 进程内所有线程的 CPU 使用情况，找到消耗 CPU 最高的那个线程的 PID (例如 5678)。
