第1题：DOCTYPE（文档类型）的作用是什么？(5分)

考察意图： 考察对HTML文档声明的基础理解



问题： 请说明<!DOCTYPE>的作用，如果不写会有什么后果？



答案：

<!DOCTYPE>的作用是告诉浏览器当前HTML文档使用哪个版本的HTML标准来解析页面。没有它，浏览器可能会进入"怪异模式"，渲染行为就跟标准不一致了。



现代网页基本都用<!DOCTYPE html>，这行代码最顶上，一个字母都不能错。它不是HTML标签，浏览器看到它就知道接下来要用HTML5的规则处理文档。



如果不写DOCTYPE，像IE6/7那种老时代，没DOCTYPE就直接怪异模式，宽度计算都能差几十像素，页面直接错乱。

第2题：HTML5相比HTML4有哪些更新？(5分)

考察意图： 考察对HTML5新特性的了解



问题： 请列举HTML5相比HTML4的主要更新点。



答案：

离线能力提升：通过Service Worker配合Cache API，可以实现离线访问，这是PWA的基础。



兼容性处理更明确：HTML5容错性强，老浏览器虽然不识别新标签，但可以通过JS创建这些元素来触发样式渲染。

第3题：src和href属性的区别是什么？(5分)

考察意图： 考察对HTML资源引入方式的理解



问题： 请说明src和href属性的区别，并举例说明。



答案：

src和href都是HTML中用来引入外部资源的属性，但它们的语义和浏览器处理方式完全不同：



src是替换行为：比如<script src="...">或<img src="...">，浏览器会去加载这个资源，并把它"放"到当前标签的位置上。你可以理解成：这个标签的内容就是由src指向的资源决定的，原始标签内容会被完全替换。



href是链接行为：通常用在<link>或<a>标签里。它不会替换当前元素，而是建立一个关联。比如<link href="style.css" rel="stylesheet">，是告诉浏览器："去加载这个CSS文件，然后应用到当前页面"，但它自己不展示内容。<a href="...">则是可点击的导航链接。



关键区别在于文档流的参与方式：



src引入的资源会阻塞渲染（比如JS），被视为页面的一部分，直接嵌入文档树



href更多是建立关联，比如样式表、图标、跳转链接，不一定会嵌入内容主体



代码示例：



html

<img src="photo.jpg"> <!-- 替换为图片 -->

<link href="theme.css" rel="stylesheet"> <!-- 关联样式，不替换 -->

<a href="/home">首页</a> <!-- 导航，用户触发 -->

简单记：src是"拿来用"，href是"挂个钩"。

第4题：title和h1标签的区别是什么？(5分)

考察意图： 考察对HTML语义化标签的理解



问题： 请说明<title>和<h1>标签的区别，在SEO中分别起什么作用？



答案：

<title>是网页的元信息，属于文档的一部分，不会直接显示在页面可视区域内。它主要出现在浏览器标签页标题、收藏夹名称或搜索引擎结果的标题位置。比如你在百度搜索一个页面，结果中显示的大标题就是来自这个页面的<title>内容。



<h1>是HTML文档中的结构化内容标签，是页面内真正展示给用户看的一级标题，通常表现为页面中最醒目的文字标题，一个页面一般建议只有一个<h1>来突出主要内容主题。



SEO中的作用：



<title>是搜索引擎最重要的参考指标，权重最高，直接影响关键词排名



<h1>也用于理解页面内容主题，但权重低于<title>



两者表达的主题应保持一致，否则可能影响SEO效果



代码示例：



html

<!DOCTYPE html>

<html>

<head>

&nbsp;   <title>这是网页标题</title>

</head>

<body>

&nbsp;   <h1>这是一级标题</h1>

</body>

</html>

简单说，<title>是给浏览器和搜索引擎看的，而<h1>是给访问者看的。

第5题：Vue中:class和:style有几种表示方式？(5分)

考察意图： 考察Vue动态样式绑定的掌握



问题： 请说明Vue中:class和:style的几种表示方式，并举例说明。



答案：

:class和:style本质是Vue对DOM class和style属性的动态绑定，核心在于支持灵活的数据驱动写法。



1\. :class的三种主流写法：



对象语法：布尔值控制开关



html

<div :class="{ active: isActive, 'text-danger': hasError }"></div>

数组语法：适合类名动态切换



html

<div :class="\[activeClass, errorClass]"></div>

混合用法：对象和数组结合，应对复杂条件



html

<div :class="\[{ active: isActive }, errorClass]"></div>

2\. :style的两种写法：



对象语法：直接内联样式，推荐驼峰式



html

<div :style="{ color: activeColor, fontSize: fontSize + 'px' }"></div>

数组语法：支持多组样式合并，常用于组件库封装



html

<div :style="\[baseStyles, overridingStyles]"></div>

Vue会自动处理浏览器前缀，比如transform自动加-webkit-等。



注意点： 频繁切换的样式建议写在CSS类中，用:class控制，而不是反复改:style对象，避免性能问题。内联样式更适合那些无法预知、运行时计算的值。

第6题：Vue中如何实现CSS样式只作用于当前组件？(5分)

考察意图： 考察对Vue作用域样式的理解



问题： 在Vue组件中，怎么使CSS样式只在当前组件中生效？原理是什么？



答案：

在Vue组件里让CSS样式只作用于当前组件，靠的是作用域样式。只需要在<style>标签上加上scoped属性，Vue就会自动处理样式隔离。



原理：

加了scoped之后，Vue编译模板时会给当前组件的每个元素都添加一个唯一的属性，比如data-v-f3f4394e，然后把样式选择器也改写成带这个属性的形式，确保样式只匹配当前组件内的元素。



示例：



html

<template>

&nbsp; <div class="example">Hello</div>

</template>



<style scoped>

.example {

&nbsp; color: red;

}

</style>

会被编译成类似：



css

.example\[data-v-f3f4394e] {

&nbsp; color: red;

}

注意事项：



如果需要影响到子组件的内部元素，得用:deep()包一层



css

:deep(.el-button) {

&nbsp; margin: 10px;

}

组件中有v-html或动态插入的内容，不会自动带上作用域属性，样式可能不受scoped控制，需要额外处理

第7题：Vue的v-show和v-if有什么区别？使用场景分别是什么？(5分)

考察意图： 考察Vue条件渲染的理解和场景判断能力



问题： 请说明Vue中v-show和v-if的区别，什么场景下使用哪个更合适？



答案：

v-show和v-if都是用来控制元素显示隐藏的指令，但实现方式完全不同：



v-if 是"真正"的条件渲染，它会根据表达式真假决定是否创建或销毁DOM节点。v-if="visible"为false时，元素压根不经过渲染，DOM树里根本不存在这个节点，切换时要重新执行组件的挂载或销毁流程。



v-show 则简单粗暴得多，不管条件如何，元素始终会被渲染到DOM中，只是通过CSS的display: none来控制视觉上的显隐。所以它的初始渲染开销更大一点，但切换成本极低，没有DOM重建过程。



使用场景：



v-if适合场景：切换不频繁、初始化性能要求高。比如用户权限不同展示完全不同的模块，用v-if可以避免无用的DOM占用内存。



v-show适合场景：需要频繁切换显示状态。比如一个常驻的弹窗或标签页，反复开关时用v-show能避免重复渲染，性能更好。



注意事项：



v-show不支持<template>标签，也不能和v-else搭配使用



用在组件上，v-if会触发整个子组件的卸载与重建，而v-show只是传个prop控制内部display

第8题：script标签中defer和async有什么区别？(5分)

考察意图： 考察对脚本加载执行机制的理解



问题： 请说明<script>标签中defer和async属性的区别，并说明各自适用场景。



答案：

defer和async都能异步加载脚本，避免阻塞页面解析，但执行时机完全不同：



async：加载完就执行，不保证顺序，跟script的位置没关系。比如写了两个async脚本，先写的可能后执行，完全看谁先下载完。这种适合独立脚本，像Google Analytics这类埋点代码，不依赖其他脚本，也不被别人依赖。



defer：等到HTML文档解析完了，DOMContentLoaded事件触发之前按顺序执行。多个defer脚本会严格按照它们在文档中出现的顺序执行，适合需要操作DOM或有依赖关系的场景。



如果同时设置：现代浏览器会优先遵循标准行为：支持defer的情况下，async会被忽略，表现得像只有defer。所以别同时写，容易自己搞混。



没有defer或async的script会阻塞HTML解析，一旦遇到就停下等脚本下载执行完再继续，这也是为什么老式代码常放在body底部。



总结：



async：下载不阻塞解析，下载完成立刻执行，执行时会阻塞解析



defer：下载不阻塞解析，等到解析完成后、DOMContentLoaded前按序执行

第9题：HTTP状态码200、301、302、400、401、403、404、500的含义是什么？(5分)

考察意图： 考察对HTTP协议基础的理解



问题： 请解释以下HTTP状态码的含义：200、301、302、400、401、403、404、500。



答案：

1xx：临时响应，比如100 Continue是客户端发请求时先问一句"我接下来要发数据，你准备好接收了吗"，服务器回100就代表可以继续。



200：最熟的，请求成功，服务端把数据正常返回。



201：资源创建成功，比如POST提交一个用户，数据库真加进去了。



204：成功处理了请求但没内容返回，像DELETE操作后常用。



301：永久重定向，比如老域名彻底不用了，搜索引擎会更新索引。



302：临时跳转，比如登录后跳回原页面。现在还有307、308，它们保证重定向时请求方法不变，比如POST不会变成GET。



400：客户端发的数据不对，比如JSON格式错了。



401：没登录，需要认证，比如没带token。



403：登录了但没权限，比如普通用户访问管理员接口。



404：大家都懂，路径写错了或者资源不存在。



500：服务器自己出问题了，代码抛异常没处理。



502：网关错误，比如Nginx转发请求给下游服务，结果那个服务挂了。



503：服务暂时不可用，可能是过载或维护。



评分标准：

3-4分：能说出5-7个状态码的含义



1-2分：只能说出3-4个



0分：几乎不知道



第10题：从输入网址到页面显示，发生了什么？(5分)

考察意图： 考察对网络请求全流程的理解



问题： 从网络角度来看，用户从输入网址到网页显示，期间发生了什么？请详细描述。



答案：

用户敲下回车后，整个过程可以分为以下几个步骤：



1\. DNS解析

浏览器先要拿到服务器IP地址。本地查hosts文件，没找到就发DNS查询请求。这个请求通常先到本地DNS服务器，它会递归或迭代查询根域名、顶级域、权威域名服务器，最终拿到A记录。



2\. TCP连接

有了IP，浏览器发起TCP连接，三次握手建立连接。如果走HTTPS，接着进行TLS握手，协商加密套件、交换密钥，确保后续通信安全。



3\. 发送HTTP请求

TCP连接建立后，浏览器开始发HTTP请求。比如GET /index.html，携带Host、User-Agent等头信息。请求通过操作系统网络栈，经过路由器层层转发，最终到达目标服务器。



4\. 服务器处理请求

服务器收到请求，应用层如Nginx或Tomcat处理，可能访问数据库、缓存，生成响应内容。响应包含状态码、响应头和HTML内容，通过已建立的TCP连接返回。



5\. 浏览器解析渲染

浏览器收到响应，解析HTML，遇到CSS、JS、图片等资源，再发起多个并行请求。这些请求可能复用现有TCP连接（HTTP Keep-Alive），减少握手开销。



页面渲染同时进行，构建DOM树、CSSOM，生成渲染树，布局绘制。JS执行可能阻塞解析，所以一般建议异步加载。



6\. 关键点说明

整个过程涉及多层协议协作。DNS查询可能耗时几百毫秒，CDN和DNS缓存能显著降低延迟。现代浏览器对同域名并发连接数限制在6个左右，超出的请求排队。



整个链路中任意环节超时或失败都会导致页面加载异常，比如DNS污染、TCP重传、TLS协商失败、后端服务无响应等。

0分：完全不知道



第11题：Cookie、Session、Token之间的区别是什么？(5分)

考察意图： 考察对用户认证机制的理解



问题： 请说明Cookie、Session、Token三者之间的区别，以及各自的优缺点。



答案：

这三个东西都是为了解决HTTP无状态的问题，但思路完全不同：



Cookie



位置：浏览器端的一小块文本，服务器通过Set-Cookie响应头下发，浏览器自动带上Cookie请求头回传



风险：存在客户端，能被JS读写（除非HttpOnly），所以有XSS风险



用途：一般用来存session id或用户偏好



优点：可以持久化



缺点：大小限制（4KB），每次请求都会携带，增加流量



Session



位置：服务器端的存储机制。用户登录后，服务端生成一个session对象，用唯一sessionId当key存在内存或Redis里



传递：sessionId通常通过Cookie传给浏览器



优点：状态集中管理，登出可以直接销毁



缺点：扩容麻烦，得搞session共享；分布式系统需要集中存储



Token（如JWT）



位置：无状态方案。用户登录成功，服务器签发一个Token，里面包含了用户信息和签名。客户端自己存，每次请求放Header里



验证：服务器不存任何状态，每次靠验签解析数据



优点：扩展性好，适合分布式；无状态



缺点：Token一旦签发没法主动作废，除非加黑名单或控制有效期很短



JWT结构：header.payload.signature，用点分割三段Base64



选型建议：单体系统用Session最省事，微服务基本都上Token。

第12题：TCP和UDP有什么区别？(5分)

考察意图： 考察对传输层协议的理解



问题： 请说明TCP和UDP的区别，各自适用于什么场景？



答案：

TCP和UDP的本质区别在于，一个追求可靠交付，一个追求极致效率：



TCP（传输控制协议）



连接：会建立连接，通过三次握手确认双方收发能力没问题



可靠性：数据包丢了会重传，顺序乱了会重新排序，保证收到的数据和发送的一致



头部：至少20字节



应用场景：HTTP、数据库连接、文件传输这些不能出错的场景



优缺点：可靠，但维护成本高，并发百万连接对服务器资源消耗大



UDP（用户数据报协议）



连接：完全不握手，发出去就完事，不管对方听没听到



可靠性：没有重传、没有流量控制



头部：只有8字节，头部开销小一半以上



应用场景：音视频通话（WebRTC）、DNS查询、DHCP分配IP、广播/组播



优缺点：性能高、延迟低，但可能丢包



代码示例对比：



java

// UDP发送：扔出去就不管了

DatagramSocket socket = new DatagramSocket();

用索引当key看着省事，但会埋坑，尤其在列表有插入、删除或排序操作时。

React的diff算法靠key来识别元素身份。如果用index作为key，比如渲染一个数组，删掉第一个元素后，后面所有项的index都变了。React以为是后面的每一项都"变"了，于是重新创建所有组件实例，触发不必要的re-render和DOM重建。更糟的是，如果有内部状态（比如input输入框），这些状态也会跟着错位。

javascript

// 正确做法

list.map(item => <div key={item.id}>{item.name}</div>)

具体问题演示：

比如原来\[A, B, C]，A对应index=0，删掉A后变成\[B, C]，B变成了index=0。React认为原来的第一个节点内容从A变成了B，第二个从B变成C，第三个被删了——它不会复用节点，而是打补丁式更新，性能差还可能出bug。



总结：



列表顺序可能变化时不要用index



数据没有唯一id时可以考虑生成一个（比如用uuid或组合字段）



静态列表、纯展示且无增删改场景下，用index问题不大

第16题：React中useState的作用是什么？(5分)

答案：

传对象也行，但要注意：它不会自动合并属性，更新对象得手动展开

答案：

计算属性和方法看着都能返回值，但底层设计目标完全不同：



computed（计算属性）



特点：基于响应式依赖进行缓存。只要依赖的数据没变，多次访问computed属性会直接返回缓存结果，不会重新执行函数

性能：性能优势明显，只有依赖数据更新时才重新计算



methods（方法）



特点：就是个普通函数，每次调用必然执行，不带任何缓存机制。在模板里调5次methods，它就执行5次，哪怕参数和数据都没变



适用场景：适合处理事件回调、发起网络请求这类不需要缓存、每次都要执行的操作



示例对比：



javascript

export default {

&nbsp; data() {

&nbsp;   return {

&nbsp;     firstName: 'John',

&nbsp;     lastName: 'Doe'

&nbsp;   }

&nbsp; },

&nbsp; computed: {

&nbsp;   fullName() {

&nbsp;     // 只有firstName或lastName改变时才重新执行

&nbsp;     return this.firstName + ' ' + this.lastName

&nbsp;   }

&nbsp; },

&nbsp; methods: {

&nbsp;   getFullName() {

&nbsp;     // 每次调用都执行

&nbsp;     return this.firstName + ' ' + this.lastName

&nbsp;   }

&nbsp; }

}

总结：别把耗时计算扔methods，尤其在模板里频繁使用时，很容易搞垮渲染性能。computed才是做这种事的正确姿势。

第20题：Vue的created和mounted生命周期钩子有什么区别？(5分)

考察意图： 考察对Vue生命周期的理解



问题： 请说明Vue中created和mounted生命周期钩子的区别，异步数据请求一般放在哪个钩子？为什么？



答案：

Vue实例的生命周期里，created和mounted是两个关键阶段，它们的触发时机和适用场景完全不同：



created阶段



时机：实例完成数据观测、属性和方法的运算，响应式系统已经建立



DOM：此时$el还没生成，页面上的DOM节点压根不存在



适用：可以在这里发起数据请求、初始化非DOM依赖的状态，比如调用接口拉取列表数据，或者设置定时器



优点：能尽早拿到数据，页面一渲染就有内容，用户体验更好



mounted阶段



时机：Vue已经把编译好的模板挂载到页面上，$el正式指向真实DOM元素



DOM：可以安全地访问和操作DOM



适用：拿DOM元素，或者初始化依赖DOM的第三方库（像ECharts、Swiper这类）



注意：服务端渲染时，mounted在客户端才会执行

一般在created钩子中请求异步数据，因为这个时候实例已经完成了数据观测、属性和方法的运算，但还没开始挂载DOM。这时候发请求，能尽早拿到数据，页面一渲染就有内容。

例外情况：如果接口依赖DOM元素（像需要获取某个节点的尺寸），那就得放到mounted里，因为created的时候DOM还没生成。



总结：数据逻辑放created，DOM交互放mounted。搞混了就可能拿到null的元素，或者在不该发请求的地方重复调用。
