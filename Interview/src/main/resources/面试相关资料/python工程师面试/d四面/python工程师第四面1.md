我理性看待加班这个问题：



【我的观点】

1\. 项目紧急时的加班是必要的，也是职业责任。

&nbsp;  比如大促前系统压测、线上故障处理、关键项目上线等，作为技术人员责无旁贷。

&nbsp;  我之前在双11期间，连续两周每天工作12小时保障系统稳定。



2\. 但加班不能是常态，也不应该是文化。

&nbsp;  长期加班说明资源规划不合理、管理有问题或者效率低下，需要反思和优化。

&nbsp;  一个成熟的团队应该靠流程和效率，而不是靠堆时间。



3\. 关键不是加不加班，而是为什么加班、加多久、有没有价值。

&nbsp;  如果是临时突发情况，可以接受；如果是长期常态，需要和leader沟通。

我分享一个最近遇到的挑战，是在双11前压测时发现的严重性能问题。



【挑战背景】

去年双11前，我们的订单系统做压力测试，发现一个严重问题：

当并发达到5000 QPS时，订单状态更新接口的P99延迟从50ms飙升到2秒，

部分请求直接超时失败。距离双11只剩2周。



【挑战难点】

1\. 时间紧：2周内必须解决，不然双11会出大事故

2\. 影响大：双11预估峰值1万QPS，这个问题不解决系统会崩

3\. 原因不明：监控看CPU、内存都不高，不知道瓶颈在哪



【解决过程】



第一阶段：定位问题（3天）



1\. 我用链路追踪工具SkyWalking分析，发现耗时主要集中在数据库更新操作

2\. 查数据库慢日志，发现大量`UPDATE orders SET status=? WHERE id=?`的慢查询

3\. 分析执行计划，发现`status`字段没索引，导致更新时锁范围大

4\. 进一步排查，发现多个事务同时更新同一个用户的不同订单，形成锁等待链

5\. 最终定位：索引缺失 + 锁竞争 + 事务过大

我提出三个优化方案，和团队讨论后决定组合使用：



方案A：加索引（简单，但治标不治本）

```sql

方案B：优化事务，减少锁持有时间

技术压力：不知道原因，白天查不到任何异常，晚上又准时出现

第一次被叫醒时，我很烦躁，心想“怎么又来了”。

第二天到公司，我马上和团队同步情况，不一个人扛。

连接池：没问题

优化SQL，加复合索引，执行时间从5分钟降到2秒

加监控，如果CPU超过80%立即告警，不用等半夜

问题原因：数据量增长导致慢SQL变极慢
